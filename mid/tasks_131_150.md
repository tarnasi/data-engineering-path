## Task 131: Semantic Layer — One Source of Truth for Metrics

### Business Context
NovaMart has a recurring problem: Marketing says revenue was $4.2M last month; Finance says $3.9M. The difference? Marketing includes pending orders, Finance excludes refunds. Every team defines "revenue" differently in their own SQL. Priya says: "Build a semantic layer. One definition per metric, exposed to every tool — dashboards, APIs, notebooks."

### Objective
Design and implement a semantic layer that provides a single, governed source of truth for NovaMart's business metrics, accessible from SQL, dashboards, and APIs.

### Technical Requirements
- **Tools**: dbt Metrics (MetricFlow) or Cube.js, Snowflake, YAML
- **Skills**: Semantic layers, metric definitions, dimensions, granularity, metric governance

### Step-by-Step Guidance
1. **Audit existing metric definitions** across teams:
   ```markdown
   ## Metric Audit: "Revenue"

   | Team | Definition | SQL | Result (June) |
   |------|-----------|-----|---------------|
   | Marketing | All orders including pending | SUM(total_amount) | $4.2M |
   | Finance | Confirmed orders minus refunds | SUM(CASE WHEN status='confirmed' THEN total_amount END) - refunds | $3.9M |
   | Product | GMV (gross merchandise value) | SUM(line_total) before discounts | $4.8M |
   | Exec Dashboard | "Revenue" (undefined) | Copy-pasted from old report | $4.0M |

   → Four different numbers for "revenue" — none wrong, all confusing
   ```

2. **Define canonical metrics in MetricFlow** (dbt Metrics):

## Task 132: Complex Event Processing (CEP)

### Business Context
NovaMart's fraud team manually reviews transactions flagged by simple rules (amount > $1000). They miss patterns across multiple events — like a user placing 5 orders from 5 different IPs in 10 minutes, or a user whose average order value suddenly triples. Priya says: "Build a complex event processing pipeline. We need to detect multi-event patterns in real-time."

### Objective
Build a CEP pipeline that detects complex, multi-event patterns in NovaMart's event streams — fraud patterns, engagement patterns, and operational anomalies.

### Technical Requirements
- **Tools**: Flink CEP or custom Python + Kafka, Redis (for state)
- **Skills**: Pattern matching on event streams, temporal constraints, stateful rules, alerting

### Step-by-Step Guidance
1. **Define event patterns to detect**:
   ```yaml
   patterns:
     - name: rapid_multi_order
       description: "User places 5+ orders within 10 minutes"
       events: OrderCreated
       condition: "COUNT(order) >= 5 WITHIN 10 MINUTES WHERE user_id = same"
       severity: high
       action: flag_fraud_review

     - name: sudden_value_spike
       description: "Order amount > 3x user's trailing 30-day average"
       events: OrderCreated
       condition: "order.total_amount > 3 * user.avg_order_value_30d"
       severity: medium
       action: flag_fraud_review

     - name: multi_ip_orders
       description: "Same user, 3+ distinct IPs within 1 hour"
       events: OrderCreated
       condition: "COUNT(DISTINCT ip_address) >= 3 WITHIN 1 HOUR WHERE user_id = same"
       severity: high
       action: block_and_alert

     - name: cart_abandonment_sequence
       description: "User adds to cart → visits checkout → leaves without purchase (within 30 min)"
       events: [AddToCart, CheckoutStart, ~Purchase]
       condition: "AddToCart FOLLOWED BY CheckoutStart NOT FOLLOWED BY Purchase WITHIN 30 MINUTES"
       severity: low
       action: trigger_marketing_email

     - name: inventory_depletion
       description: "Product sells 10x normal daily rate within 1 hour"
       events: OrderCreated
       condition: "COUNT(product_id = X) > 10 * daily_avg WITHIN 1 HOUR"
       severity: medium
       action: alert_ops_team
   ```

2. **Flink CEP implementation**:
   ```sql
   -- Flink SQL: detect rapid multi-order pattern
   SELECT *
   FROM order_events
   MATCH_RECOGNIZE (
       PARTITION BY user_id
       ORDER BY event_time
       MEASURES
           FIRST(A.event_time) AS pattern_start,
           LAST(A.event_time) AS pattern_end,
           COUNT(A.order_id) AS order_count,
           SUM(A.total_amount) AS total_amount
       ONE ROW PER MATCH
       AFTER MATCH SKIP PAST LAST ROW
       PATTERN (A{5,})
       DEFINE
           A AS A.event_time - FIRST(A.event_time) < INTERVAL '10' MINUTE
   );
   ```

3. **Python CEP engine** (for teams without Flink):
   ```python
   from collections import defaultdict
   from datetime import datetime, timedelta

   class CEPEngine:
       def __init__(self):
           self.patterns = []
           self.state = defaultdict(list)  # user_id -> event buffer

       def register_pattern(self, pattern: Pattern):
           self.patterns.append(pattern)

       def process_event(self, event: dict):
           user_id = event['user_id']
           self.state[user_id].append(event)

           # Expire old events (sliding window)
           cutoff = datetime.utcnow() - timedelta(hours=1)
           self.state[user_id] = [
               e for e in self.state[user_id]
               if e['timestamp'] > cutoff
           ]

           # Check all patterns
           for pattern in self.patterns:
               match = pattern.evaluate(self.state[user_id])
               if match:
                   self.fire_alert(pattern, match, user_id)

   class RapidMultiOrderPattern:
       def evaluate(self, events: list) -> Optional[dict]:
           orders = [e for e in events if e['type'] == 'OrderCreated']
           window = timedelta(minutes=10)

           # Sliding window: any 10-min window with 5+ orders?
           for i, event in enumerate(orders):
               window_events = [
                   e for e in orders[i:]
                   if e['timestamp'] - event['timestamp'] <= window
               ]
               if len(window_events) >= 5:
                   return {
                       'pattern': 'rapid_multi_order',
                       'events': window_events,
                       'start': event['timestamp'],
                       'end': window_events[-1]['timestamp'],
                   }
           return None
   ```

4. **State management with Redis** (for distributed state):
   ```python
   class DistributedCEPState:
       def __init__(self):
           self.redis = redis.Redis()

       def add_event(self, user_id: int, event: dict):
           key = f"cep:events:{user_id}"
           self.redis.lpush(key, json.dumps(event))
           self.redis.expire(key, 3600)  # 1-hour TTL

       def get_events(self, user_id: int, window_minutes: int) -> list:
           key = f"cep:events:{user_id}"
           all_events = [json.loads(e) for e in self.redis.lrange(key, 0, -1)]
           cutoff = datetime.utcnow() - timedelta(minutes=window_minutes)
           return [e for e in all_events if parse(e['timestamp']) > cutoff]
   ```

5. **Alert routing**: fraud → PagerDuty, cart abandonment → Marketing API, inventory → Ops Slack

6. **Testing**: produce synthetic event sequences that match and don't match each pattern, verify detection

### Expected Output
- Pattern definitions: `tasks/132_cep/patterns.yaml`
- CEP engine: `tasks/132_cep/cep_engine.py`
- Flink CEP queries: `tasks/132_cep/flink/`
- Alert router: `tasks/132_cep/alerts.py`

### Common Mistakes
- Unbounded state — events accumulate forever; set TTLs and window bounds
- Patterns too sensitive — too many false positives overwhelm the fraud team
- Not testing edge cases — overlapping patterns, events at exact window boundaries
- Alerting without deduplication — same pattern firing 100 alerts in 1 minute

## Task 133: Schema Evolution at Scale

### Business Context
NovaMart's platform now ingests data from 12 different sources. Schema changes happen frequently — a new column added by the commerce team, a field renamed by QuickShop's legacy system adapter, a data type changed by the payments provider. Every schema change risks breaking downstream pipelines. Priya says: "Build a schema evolution strategy that lets sources change without breaking consumers."

### Objective
Design and implement a comprehensive schema evolution strategy — forward and backward compatibility, schema registry, migration automation, and consumer impact analysis.

### Technical Requirements
- **Tools**: Confluent Schema Registry (Avro), dbt, Python, Snowflake
- **Skills**: Schema compatibility, Avro/Protobuf, schema registries, forward/backward compatibility, migration automation

### Step-by-Step Guidance
1. **Understand compatibility levels**:
   ```markdown
   ## Schema Compatibility Matrix

   | Change | Backward | Forward | Full |
   |--------|----------|---------|------|
   | Add optional field | ✅ | ✅ | ✅ |
   | Add required field | ❌ | ✅ | ❌ |
   | Remove optional field | ✅ | ❌ | ❌ |
   | Remove required field | ❌ | ❌ | ❌ |
   | Rename field | ❌ | ❌ | ❌ |
   | Change type (int→string) | ❌ | ❌ | ❌ |
   | Add enum value | ✅ | ❌ | ❌ |

   NovaMart policy: BACKWARD compatible for all topics
   → New consumers can read old data; old consumers can read new data
   ```

2. **Build the DAG factory**:
   ```python
   # dags/factory/dag_factory.py
   import yaml
   from pathlib import Path
   from airflow import DAG
   from datetime import datetime, timedelta

   def create_dag_from_config(config_path: str) -> DAG:
       with open(config_path) as f:
           config = yaml.safe_load(f)['pipeline']

       dag = DAG(
           dag_id=config['name'],
           schedule_interval=config['schedule'],
           default_args={
               'owner': config['owner'],
               'retries': 2,
               'retry_delay': timedelta(minutes=5),
               'sla': timedelta(hours=config.get('sla_hours', 4)),
           },
           tags=config.get('tags', []),
           catchup=False,
           start_date=datetime(2024, 1, 1),
       )

       with dag:
           # Build source extraction tasks
           extract_tasks = build_extract_tasks(config['source'])

           # Build quality check tasks
           pre_quality = build_quality_tasks(config['quality'], prefix='pre')

           # Build transform tasks
           transform_tasks = build_transform_tasks(config['transform'])

           # Build post-transform quality
           post_quality = build_quality_tasks(config['quality'], prefix='post')

           # Wire dependencies
           extract_tasks >> pre_quality >> transform_tasks >> post_quality

       return dag

   def build_extract_tasks(source_config: dict) -> list:
       tasks = []
       if source_config['type'] == 'postgres_cdc':
           for table in source_config['tables']:
               task = SnowflakeCopyOperator(
                   task_id=f"extract_{table['name']}",
                   snowflake_conn_id='snowflake_prod',
                   stage_path=f"novamart_s3_stage/{table['name']}/",
                   target_table=f"raw.{table['name']}",
               )
               tasks.append(task)
       return tasks

   def build_transform_tasks(transform_config: dict) -> list:
       if transform_config['type'] == 'dbt':
           return [DbtRunOperator(
               task_id='dbt_run',
               project=transform_config['project'],
               models=' '.join(transform_config['models']),
               target=transform_config['target'],
           )]
       return []
   ```

3. **Auto-discover and register DAGs**:
   ```python

   ```

## Task 135: Flink State Machines for Order Lifecycle

### Business Context
NovaMart needs to track order lifecycle state transitions in real-time: Created → Confirmed → Shipped → Delivered (or Cancelled at any point). The current system only knows the latest state. The operations team needs: "How long does each state transition take? Are orders stuck in 'Confirmed' for too long? What's the drop-off rate between 'Shipped' and 'Delivered'?"

### Objective
Build a stateful stream processing pipeline using Flink (or Spark) that models the order lifecycle as a state machine, tracking transitions, timing, and anomalies in real-time.

### Technical Requirements
- **Tools**: Flink (Java/Python) or Spark Structured Streaming, Kafka
- **Skills**: State machines, stateful stream processing, timers, side outputs, anomaly detection

### Step-by-Step Guidance
1. **Define the state machine**:
   ```
   ┌──────────┐     ┌───────────┐     ┌──────────┐     ┌───────────┐
   │ Created  │────→│ Confirmed │────→│ Shipped  │────→│ Delivered │
   └──────────┘     └───────────┘     └──────────┘     └───────────┘
        │                │                  │
        └────────────────┴──────────────────┘
                         │
                    ┌────┴─────┐
                    │ Cancelled│
                    └──────────┘

   Valid transitions:
   - Created → Confirmed
   - Created → Cancelled
   - Confirmed → Shipped
   - Confirmed → Cancelled
   - Shipped → Delivered
   - Shipped → Cancelled (return/lost)

   Invalid transitions (anomalies):
   - Delivered → anything
   - Shipped → Created (backward)
   - Any state → same state (duplicate event)
   ```

2. **Flink state machine implementation**:
   ```python
   # Using Flink's KeyedProcessFunction
   class OrderStateMachine(KeyedProcessFunction):
       VALID_TRANSITIONS = {
           None: {'Created'},
           'Created': {'Confirmed', 'Cancelled'},
           'Confirmed': {'Shipped', 'Cancelled'},
           'Shipped': {'Delivered', 'Cancelled'},
           'Delivered': set(),  # terminal state
           'Cancelled': set(),  # terminal state
       }

       SLA_LIMITS = {
           ('Created', 'Confirmed'): timedelta(hours=2),
           ('Confirmed', 'Shipped'): timedelta(days=3),
           ('Shipped', 'Delivered'): timedelta(days=7),
       }

       def open(self, runtime_context):
           self.state = runtime_context.get_state(ValueStateDescriptor('order_state', Types.STRING()))
           self.entered_at = runtime_context.get_state(ValueStateDescriptor('entered_at', Types.LONG()))

       def process_element(self, event, ctx):
           current_state = self.state.value()
           new_state = event['status']

           # Check valid transition
           if new_state not in self.VALID_TRANSITIONS.get(current_state, set()):
               ctx.output(ANOMALY_TAG, {
                   'order_id': event['order_id'],
                   'invalid_transition': f"{current_state} → {new_state}",
                   'event': event,
               })
               return

           # Calculate transition time
           if current_state and self.entered_at.value():
               transition_time = event['timestamp'] - self.entered_at.value()
               transition_key = (current_state, new_state)

               ctx.output(TRANSITION_TAG, {
                   'order_id': event['order_id'],
                   'from_state': current_state,
                   'to_state': new_state,
                   'duration_seconds': transition_time,
               })

               # Check SLA
               sla = self.SLA_LIMITS.get(transition_key)
               if sla and transition_time > sla.total_seconds():
                   ctx.output(SLA_BREACH_TAG, {
                       'order_id': event['order_id'],
                       'transition': transition_key,
                       'actual': transition_time,
                       'sla': sla.total_seconds(),
                   })

           # Update state
           self.state.update(new_state)
           self.entered_at.update(event['timestamp'])

           # Set timer for SLA monitoring
           sla = self.SLA_LIMITS.get((new_state, None))
           if sla:
               ctx.timer_service().register_event_time_timer(
                   event['timestamp'] + sla.total_seconds() * 1000
               )

       def on_timer(self, timestamp, ctx):
           # Timer fired — order has been in current state too long
           current = self.state.value()
           if current not in ('Delivered', 'Cancelled'):
               ctx.output(STUCK_ORDER_TAG, {
                   'order_id': ctx.get_current_key(),
                   'stuck_in': current,
                   'since': self.entered_at.value(),
               })
   ```

   ```python
               'braze': BrazeDestination(),
               'algolia': AlgoliaDestination(),
               'google_ads': GoogleAdsDestination(),
           }

       def run_sync(self, sync_config: dict):
           # 1. Extract from warehouse
           source_data = self.extract(sync_config['source'])

           # 2. Detect changes (only sync what's new/changed)
           changes = self.detect_changes(sync_config['name'], source_data)

           # 3. Transform (apply field mapping)
           mapped_data = self.apply_mapping(changes, sync_config['destination']['mapping'])

           # 4. Load to destination (with rate limiting and batching)
           destination = self.destinations[sync_config['destination']['type']]
           result = destination.sync(mapped_data)

           # 5. Record sync state
           self.record_sync(sync_config['name'], result)
           return result

       def detect_changes(self, sync_name: str, current_data: pd.DataFrame) -> pd.DataFrame:
           """Compare with last sync snapshot to find additions, updates, deletions"""
           last_snapshot = self.get_last_snapshot(sync_name)
           if last_snapshot is None:
               return current_data  # first sync: everything is new

           added = current_data[~current_data['user_id'].isin(last_snapshot['user_id'])]
           removed = last_snapshot[~last_snapshot['user_id'].isin(current_data['user_id'])]
           updated = self.find_updated_rows(current_data, last_snapshot)

           return pd.concat([
               added.assign(_change_type='add'),
               updated.assign(_change_type='update'),
               removed.assign(_change_type='delete'),
           ])
   ```

   ```python
           batches = [data[i:i+self.BATCH_SIZE] for i in range(0, len(data), self.BATCH_SIZE)]
           results = SyncResult()

           for batch in batches:

                   'attributes': [
                       {
                           'external_id': str(row['external_id']),
                           **{k: v for k, v in row.items() if k.startswith('custom_attribute.')}
   ```

   ```python
   class ContractValidator:
       def validate(self, data: pd.DataFrame, contract: dict) -> ValidationResult:
           results = ValidationResult(contract['contract']['name'])

           # Schema checks
           for col_spec in contract['schema']['columns']:
               col_name = col_spec['name']

               # Column existence
               if col_name not in data.columns:
                   results.add_failure('schema', f"Missing column: {col_name}")
                   continue

               # Type check
               if not self.check_type(data[col_name], col_spec['type']):
                   results.add_failure('schema', f"{col_name}: expected {col_spec['type']}")

               # Nullable check
               if not col_spec.get('nullable', True):
                   null_count = data[col_name].isna().sum()
                   if null_count > 0:
                       results.add_failure('nullable', f"{col_name}: {null_count} nulls (not nullable)")

               # Allowed values
               if 'allowed_values' in col_spec:
                   invalid = data[~data[col_name].isin(col_spec['allowed_values']) & data[col_name].notna()]
                   if len(invalid) > 0:
                       results.add_failure('allowed_values',
                           f"{col_name}: {len(invalid)} rows with invalid values")

               # Constraints
               if 'constraints' in col_spec:
                   self.check_constraints(data[col_name], col_spec['constraints'], col_name, results)

           # Quality checks
           quality = contract.get('quality', {})
           row_count = len(data)
           if row_count < quality.get('row_count', {}).get('min', 0):
               results.add_failure('quality', f"Row count {row_count} below minimum")
           if row_count > quality.get('row_count', {}).get('max', float('inf')):
               results.add_failure('quality', f"Row count {row_count} above maximum")

           return results
   ```

## Task 143: Idempotent Pipeline Design Patterns

### Business Context

Every re-run risks duplicates or over-counts. Priya says: "Every pipeline must be idempotent. Run it once or 100 times — same result."

### Objective
Refactor NovaMart's key pipelines to be fully idempotent — safe to re-run at any time without producing duplicates or incorrect results.

### Technical Requirements
- **Tools**: SQL (Snowflake MERGE), Python, dbt (incremental), Airflow
- **Skills**: Idempotency patterns, MERGE/upsert, partition overwrite, deduplication, tombstones

### Step-by-Step Guidance
1. **Identify non-idempotent patterns** in existing pipelines:
   ```markdown
   ## Non-Idempotent Patterns Found

   | Pattern | Example | Problem on Re-run |
   |---------|---------|-------------------|
   | INSERT without dedup | INSERT INTO fact_orders SELECT ... | Duplicates |
   | Auto-increment IDs | INSERT INTO dim_users (id, ...) | ID collision |
   | DELETE + INSERT (gap) | DELETE WHERE date = X; INSERT | Data loss if INSERT fails |
   | Running totals | UPDATE summary SET total = total + daily_count | Double-counting |
   | Append-only logs | INSERT INTO audit_log | Duplicate log entries |
   ```

2. **Pattern 1: MERGE (upsert)** — most common fix:
   ```sql
   -- INSTEAD OF: INSERT INTO fact_orders SELECT ...
   -- USE:
   MERGE INTO analytics.fact_orders AS target
   USING staging.stg_orders AS source
       ON target.order_id = source.order_id
   WHEN MATCHED THEN UPDATE SET
       total_amount = source.total_amount,
       status = source.status,
       updated_at = CURRENT_TIMESTAMP
   WHEN NOT MATCHED THEN INSERT (order_id, user_id, total_amount, status, updated_at)
       VALUES (source.order_id, source.user_id, source.total_amount, source.status, CURRENT_TIMESTAMP);
   ```

3. **Pattern 2: Partition overwrite** — for date-partitioned tables:
   ```sql
   -- Delete-then-insert for a specific partition (atomic in Snowflake)
   -- This is idempotent: same result regardless of how many times you run it
   DELETE FROM analytics.daily_revenue WHERE date = '{{ ds }}';
   INSERT INTO analytics.daily_revenue
   SELECT
       '{{ ds }}' AS date,
       SUM(total_amount) AS revenue,
       COUNT(DISTINCT order_id) AS order_count
   FROM staging.stg_orders
   WHERE order_date = '{{ ds }}';

   -- Even better: use a transaction
   BEGIN;
   DELETE FROM analytics.daily_revenue WHERE date = '{{ ds }}';
   INSERT INTO analytics.daily_revenue SELECT ...;
   COMMIT;
   ```

4. **Pattern 3: dbt incremental with unique key**:
   ```sql
   -- models/fact_order_items.sql
   {{ config(
       materialized='incremental',
       unique_key='order_item_id',
       incremental_strategy='merge',
       on_schema_change='sync_all_columns',
   ) }}

   SELECT
       order_item_id,
       order_id,
       product_id,
       quantity,
       line_total,
       loaded_at
   FROM {{ ref('stg_order_items') }}
   {% if is_incremental() %}
   WHERE loaded_at > (SELECT MAX(loaded_at) FROM {{ this }})
   {% endif %}
   ```

5. **Pattern 4: Idempotent Python with checkpointing**:
   ```python
   class IdempotentPipeline:
       def run(self, execution_date: date):
           checkpoint = self.get_checkpoint(execution_date)

           if checkpoint.step < 1:
               self.extract(execution_date)
               self.save_checkpoint(execution_date, step=1)

           if checkpoint.step < 2:
               self.transform(execution_date)
               self.save_checkpoint(execution_date, step=2)

           if checkpoint.step < 3:
               self.load(execution_date)  # uses MERGE, not INSERT
               self.save_checkpoint(execution_date, step=3)

           # Re-running skips completed steps, re-runs from last checkpoint
   ```

6. **Pattern 5: Deterministic IDs** (no auto-increment):
   ```python
   # Instead of auto-increment, derive ID from business key
   import hashlib

   def generate_deterministic_id(order_id: int, product_id: int) -> str:
       key = f"{order_id}:{product_id}"
       return hashlib.md5(key.encode()).hexdigest()

   # Same input always generates same ID → safe to re-run
   ```

7. **Testing idempotency**: run every pipeline twice, compare outputs — they must be identical

- Idempotency patterns: `tasks/143_idempotent_pipelines/patterns/`
- Refactored pipelines: `tasks/143_idempotent_pipelines/pipelines/`

### Common Mistakes
- MERGE without a proper key — matching on wrong columns causes incorrect updates
- DELETE + INSERT without transaction — if INSERT fails, data is gone
- Using `inserted_at` as the unique key — timestamps aren't unique at high throughput

   ```sql
       COUNT(DISTINCT query_id) AS query_count,
       COUNT(DISTINCT user_name) AS unique_users,
       MAX(query_start_time) AS last_accessed,
       DATEDIFF(day, MAX(query_start_time), CURRENT_DATE) AS days_since_access
   FROM SNOWFLAKE.ACCOUNT_USAGE.ACCESS_HISTORY,
       LATERAL FLATTEN(base_objects_accessed) base_objects_accessed
   ```

   ```python
       def find_candidates(self) -> list:

               if table.unique_users == 0:
                   reasons.append("No users in last 90 days")

                       'monthly_cost': table.monthly_cost,
                       'recommendation': 'deprecate' if len(reasons) >= 2 else 'review',
   ```

## Task 147: Real-Time Data Warehouse (Streaming into Snowflake)

### Business Context
NovaMart's dashboards refresh every 6 hours (the batch pipeline cycle). The operations team needs real-time visibility: orders per minute, current inventory, live customer support metrics. ### Objective
Build a streaming ingestion pipeline that delivers near-real-time data (< 5 minute latency) to Snowflake, enabling real-time dashboards and operational reporting.

### Technical Requirements
- **Tools**: Kafka, Snowpipe Streaming (or Spark Structured Streaming → Snowflake), Snowflake Dynamic Tables
- **Skills**: Streaming to warehouse, micro-batch vs true streaming, materialized views, real-time dashboards

### Step-by-Step Guidance
1. **Choose the real-time pattern**:
   ```markdown
   | Pattern | Latency | Complexity | Cost | Best For |
   |---------|---------|-----------|------|----------|
   | Snowpipe (file-based) | 2-5 min | Low | Low | Structured data from S3 |
   | Snowpipe Streaming API | < 30 sec | Medium | Medium | Kafka events to Snowflake |
   | Spark → Snowflake micro-batch | 1-5 min | Medium | Medium | Complex transformations |
   | Dynamic Tables | Depends on source | Low | Low | Cascading real-time transforms |
   ```

2. **Snowpipe Streaming** (direct Kafka → Snowflake):
   ```python
   from snowflake.ingest import SnowflakeStreamingIngestClient, SnowflakeStreamingIngestChannel

   client = SnowflakeStreamingIngestClient(
       name='novamart_streaming_client',
       account='novamart',
       user='streaming_service',
       private_key=load_private_key(),
   )

   channel = client.open_channel(
       name='orders_channel',
       database='novamart_dw',
       schema='raw',
       table='orders_streaming',
   )

   # Read from Kafka, write to Snowflake
   for msg in kafka_consumer:
       order = json.loads(msg.value())
       channel.insert_row({
           'order_id': order['order_id'],
           'user_id': order['user_id'],
           'total_amount': order['total_amount'],
           'order_date': order['order_date'],
           'status': order['status'],
           '_ingested_at': datetime.utcnow().isoformat(),
       })

       if channel.get_latest_committed_offset_token() % 1000 == 0:
           channel.flush()  # Commit batch to Snowflake
   ```

3. **Snowflake Dynamic Tables** (real-time transformations):
   ```sql
   -- Dynamic Table: automatically refreshes when source data changes
   CREATE OR REPLACE DYNAMIC TABLE analytics.rt_orders_summary
       TARGET_LAG = '5 minutes'  -- refresh within 5 minutes of source change
       WAREHOUSE = streaming_wh
   AS
   SELECT
       DATE_TRUNC('hour', order_date) AS hour,
       COUNT(*) AS order_count,
       SUM(total_amount) AS revenue,
       COUNT(DISTINCT user_id) AS unique_customers,
       AVG(total_amount) AS avg_order_value
   FROM raw.orders_streaming
   WHERE order_date >= DATEADD(day, -7, CURRENT_DATE)
   GROUP BY 1;

   -- Chain dynamic tables: orders → enriched → aggregated
   CREATE OR REPLACE DYNAMIC TABLE analytics.rt_orders_enriched
       TARGET_LAG = '5 minutes'
       WAREHOUSE = streaming_wh
   AS
   SELECT
       o.*,
       u.segment AS customer_segment,
       p.category AS product_category
   FROM raw.orders_streaming o
   JOIN analytics.dim_users u ON o.user_id = u.user_id
   JOIN analytics.dim_products p ON o.product_id = p.product_id;
   ```

4. **Lambda architecture** — merge batch and streaming:
   ```sql
   -- Real-time view: union of batch (reliable, complete) + streaming (fresh, approximate)
   CREATE VIEW analytics.orders_realtime AS
   -- Batch data: complete, T-6 hours
   SELECT order_id, user_id, total_amount, order_date, 'batch' AS source
   FROM analytics.fact_orders  -- updated every 6 hours
   WHERE order_date < DATEADD(hour, -6, CURRENT_TIMESTAMP)
   UNION ALL
   -- Streaming data: fresher, last 6 hours
   SELECT order_id, user_id, total_amount, order_date, 'stream' AS source
   FROM raw.orders_streaming
   WHERE _ingested_at >= DATEADD(hour, -6, CURRENT_TIMESTAMP);
   ```

5. **Real-time dashboard queries**:
   ```sql
   -- Current orders per minute (last hour)
   SELECT
       DATE_TRUNC('minute', _ingested_at) AS minute,
       COUNT(*) AS orders
   FROM raw.orders_streaming
   WHERE _ingested_at >= DATEADD(hour, -1, CURRENT_TIMESTAMP)
   GROUP BY 1 ORDER BY 1;
   ```

6. **Monitoring**: streaming lag, Snowpipe status, Dynamic Table refresh schedule

### Expected Output
- Streaming pipeline: `tasks/147_realtime_warehouse/streaming/`
- Dynamic tables: `tasks/147_realtime_warehouse/dynamic_tables.sql`
- Lambda architecture: `tasks/147_realtime_warehouse/lambda_views.sql`
- Real-time dashboard queries: `tasks/147_realtime_warehouse/queries/`
- Latency measurements: end-to-end from Kafka produce to Snowflake queryable

### Common Mistakes
- Everything real-time — most analytics are fine with hourly or daily freshness; real-time adds cost
- No deduplication in streaming — events can be delivered twice; use MERGE or dedup in Dynamic Tables
- Lambda architecture without reconciliation — batch and streaming produce slightly different numbers
- Small warehouse for streaming — streaming workloads are continuous; right-size or use auto-scaling

### Stretch Goal
Build a Kappa architecture: replace the Lambda dual-path with a single streaming path that handles both real-time and backfill.
