## Task 124: Advanced Kafka — Exactly-Once Semantics

### Business Context
NovaMart's loyalty points pipeline processes order events from Kafka and adds points. Currently, if the consumer crashes after processing but before committing the offset, the event is re-processed — doubling the customer's points. Ravi says: "For financial data, at-least-once isn't good enough. We need exactly-once semantics."

### Objective
Implement exactly-once processing in Kafka: idempotent producers, transactional consumers, and the Kafka Transactions API.

### Technical Requirements
- **Tools**: Kafka (Transactions API), Python (confluent-kafka), PostgreSQL
- **Skills**: Exactly-once semantics, idempotent producers, transactions, outbox pattern, deduplication

### Step-by-Step Guidance
1. **Understand the problem— duplicate processing**:
   ```
   Normal flow:
   1. Consumer reads event (order 10001, +500 points)
   2. Consumer processes (INSERT into points table: +500)
   3. Consumer commits offset
   
   Failure scenario (at-least-once):
   1. Consumer reads event (order 10001, +500 points)
   2. Consumer processes (INSERT: +500) ✓
   3. Consumer crashes before committing offset ✗
   4. Consumer restarts, re-reads event (order 10001, +500 points)
   5. Consumer processes again (INSERT: +500 again!) — DOUBLE POINTS
   ```

2. **Solution 1: Idempotent producer** (Kafka-side):
   ```python
   producer = Producer({
       'bootstrap.servers': 'localhost:9092',
       'enable.idempotence': True,  # Kafka deduplicates retried produces
       'acks': 'all',
       'max.in.flight.requests.per.connection': 5,
       'retries': 2147483647,  # retry forever
   })
   ```

3. **Solution 2: Transactional consume-transform-produce** (Kafka Transactions API):
   ```python
   producer = Producer({
       'bootstrap.servers': 'localhost:9092',
       'transactional.id': 'loyalty-points-processor',  # enables transactions
   })
   producer.init_transactions()

   consumer = Consumer({
       'bootstrap.servers': 'localhost:9092',
       'group.id': 'loyalty-points',
       'enable.auto.commit': False,
       'isolation.level': 'read_committed',  # only read committed messages
   })

   while True:
       msg = consumer.poll(1.0)
       if msg is None:
           continue

       # Begin transaction
       producer.begin_transaction()
       try:
           # Process
           result = process_order_points(msg)

           # Produce to output topic (within transaction)
           producer.produce('novamart.loyalty.points-updated', value=result)

           # Commit consumer offset within the same transaction
           producer.send_offsets_to_transaction(
               consumer.position(consumer.assignment()),
               consumer.consumer_group_metadata()
           )

           # Commit transaction (atomically: produce + offset commit)
           producer.commit_transaction()
       except Exception:
           producer.abort_transaction()
   ```

4. **Solution 3: Application-level deduplication** (most practical for DB sinks):
   ```python
   class IdempotentPointsProcessor:
       def process(self, event: dict):
           event_id = event['event_id']

           # Check if already processed
           existing = query("SELECT 1 FROM processed_events WHERE event_id = %s", event_id)
           if existing:
               self.log.info(f"Skipping duplicate event: {event_id}")
               return

           # Process within a transaction
           with db.transaction():
               # Add points
               execute("UPDATE user_points SET balance = balance + %s WHERE user_id = %s",
                       event['points'], event['user_id'])
               # Record that we processed this event
               execute("INSERT INTO processed_events (event_id, processed_at) VALUES (%s, NOW())",
                       event_id)

           # Commit Kafka offset only after DB transaction succeeds
           consumer.commit()
   ```

5. **Solution 4: Outbox pattern** (for cross-system exactly-once):
   ```python
   # Instead of writing to DB AND producing to Kafka (two systems, can't be atomic):
   # Write to DB + outbox table in one transaction
   # A separate process reads the outbox and publishes to Kafka

   with db.transaction():
       execute("UPDATE user_points SET balance = balance + %s", points)
       execute("INSERT INTO outbox (topic, key, value) VALUES (%s, %s, %s)",
               'novamart.loyalty.points-updated', user_id, json.dumps(result))

   # Outbox publisher (separate process)
   class OutboxPublisher:
       def publish_pending(self):
           events = query("SELECT * FROM outbox WHERE published = FALSE ORDER BY created_at")
           for event in events:
               producer.produce(event.topic, key=event.key, value=event.value)
               execute("UPDATE outbox SET published = TRUE WHERE id = %s", event.id)
   ```

6. **Compare approaches**:
   | Approach | Complexity | Guarantees | Best For |
   |----------|-----------|------------|----------|
   | Idempotent producer | Low | No duplicate produces | Kafka-to-Kafka |
   | Kafka Transactions | Medium | Exactly-once Kafka-to-Kafka | Stream processing |
   | App-level dedup | Medium | Exactly-once to any sink | DB writes |
   | Outbox pattern | High | Cross-system consistency | Microservices |

### Expected Output
- Implementations: `tasks/124_exactly_once/` (one per approach)
- Comparison doc: `tasks/124_exactly_once/comparison.md`
- Test: produce 1000 events, crash consumer 10 times, verify exactly 1000 records in DB
- Decision doc: which approach for which NovaMart use case

### Common Mistakes
- Thinking Kafka's `exactly-once` means end-to-end — it's Kafka-to-Kafka; DB sinks need app-level handling
- Processed_events table growing forever — add a retention policy (delete events older than 7 days)
- Outbox table not being published — need monitoring on outbox lag
- Ignoring the performance cost — deduplication checks add latency; understand the trade-off

## Task 125: Stateful Stream Processing — Sessions and Windows

### Business Context
NovaMart's clickstream data is a fire hose: 5000 events/second from web and mobile. The ML team needs sessionized data — grouping raw click events into browsing sessions — for their models. Marketing needs real-time conversion funnels. Ravi says: "Build a streaming pipeline that sessionizes clickstream data in real-time."

### Objective
Build a stateful stream processing job that converts raw clickstream events into browsing sessions using session windows, and computes real-time conversion funnels using hopping windows.

### Technical Requirements
- **Tools**: Flink SQL or Spark Structured Streaming, Kafka
- **Skills**: Session windows, tumbling windows, hopping windows, watermarks, late data handling, state management

### Step-by-Step Guidance
1. **Define the input events**:
   ```json
   {
     "event_id": "evt-123456",
     "user_id": 42,
     "event_type": "page_view",
     "page": "/products/laptop-pro-x1",
     "category": "electronics",
     "timestamp": "2024-07-20T14:23:45Z",
     "device": "mobile",
     "referrer": "google.com"
   }
   ```

2. **Session window — group clicks into sessions** (30-min inactivity gap):
   ```sql
   -- Flink SQL
   CREATE TABLE clickstream (
       event_id STRING,
       user_id INT,
       event_type STRING,
       page STRING,
       category STRING,
       event_time TIMESTAMP(3),
       device STRING,
       WATERMARK FOR event_time AS event_time - INTERVAL '2' MINUTE
   ) WITH (
       'connector' = 'kafka',
       'topic' = 'novamart.clickstream.events',
       'format' = 'json'
   );

   -- Sessionize: group into sessions with 30-min gap
   SELECT
       user_id,
       SESSION_START(event_time, INTERVAL '30' MINUTE) AS session_start,
       SESSION_END(event_time, INTERVAL '30' MINUTE) AS session_end,
       COUNT(*) AS event_count,
       COUNT(DISTINCT page) AS pages_viewed,
       COUNT(DISTINCT category) AS categories_browsed,
       LISTAGG(page, ' -> ') AS page_path,
       MAX(CASE WHEN event_type = 'purchase' THEN 1 ELSE 0 END) AS converted
   FROM clickstream
   GROUP BY user_id, SESSION(event_time, INTERVAL '30' MINUTE);
   ```

3. **Spark Structured Streaming equivalent**:
   ```python
   sessions = clickstream \
       .withWatermark("event_time", "2 minutes") \
       .groupBy(
           col("user_id"),
           session_window("event_time", "30 minutes")
       ) \
       .agg(
           count("*").alias("event_count"),
           countDistinct("page").alias("pages_viewed"),
           min("event_time").alias("session_start"),
           max("event_time").alias("session_end"),
           collect_list("page").alias("page_path"),
           max(when(col("event_type") == "purchase", 1).otherwise(0)).alias("converted")
       )
   ```

4. **Real-time conversion funnel** (hopping window):
   ```sql
   -- 5-minute hopping window, updated every 1 minute
   SELECT
       HOP_START(event_time, INTERVAL '1' MINUTE, INTERVAL '5' MINUTE) AS window_start,
       COUNT(DISTINCT CASE WHEN event_type = 'page_view' THEN user_id END) AS viewers,
       COUNT(DISTINCT CASE WHEN event_type = 'add_to_cart' THEN user_id END) AS cart_adds,
       COUNT(DISTINCT CASE WHEN event_type = 'checkout_start' THEN user_id END) AS checkouts,
       COUNT(DISTINCT CASE WHEN event_type = 'purchase' THEN user_id END) AS purchases
   FROM clickstream
   GROUP BY HOP(event_time, INTERVAL '1' MINUTE, INTERVAL '5' MINUTE);
   ```

5. **Handle late data**:
   ```python
   # Watermark: events arriving > 2 minutes late are dropped
   # But what about really late events? (user's phone was offline)

   class LateDataHandler:
       def process(self, event: dict, watermark: datetime):
           lateness = watermark - event['event_time']
           if lateness > timedelta(minutes=2):
               # Route to late data topic for batch reprocessing
               self.producer.produce('novamart.clickstream.late', value=event)
               self.metrics.increment('late_events')
           else:
               # Normal processing
               self.process_normally(event)
   ```

6. **State management considerations**:
   ```yaml
   state_management:
     # Each user_id maintains session state
     state_per_user:
       - current_session_events: list
       - session_start: timestamp
       - last_event_time: timestamp

     # With 2M users, state can be large
     state_size_estimate: "~2M users × 1KB avg = 2GB state"

     # State backend: RocksDB (Flink) or in-memory (Spark)
     backend: rocksdb  # disk-backed for large state

     # State TTL: expire inactive sessions after 24 hours
     ttl: 24h
   ```

7. **Output**: write sessionized data to Kafka topic and Snowflake table:
   ```python
   # To Kafka for real-time consumers
   sessions.writeStream \
       .format("kafka") \
       .option("topic", "novamart.clickstream.sessions") \
       .start()

   # To Snowflake for analytics
   sessions.writeStream \
       .foreachBatch(write_to_snowflake) \
       .start()
   ```

### Expected Output
- Sessionization job: `tasks/125_stateful_streaming/sessionization.py`
- Funnel computation: `tasks/125_stateful_streaming/funnel.py`
- Late data handler: `tasks/125_stateful_streaming/late_data.py`
- Demo: stream 10,000 clickstream events, show session output
- State management design doc

### Common Mistakes
- No watermark → sessions never close, memory grows until OOM
- Session gap too short (5 min) → users switching tabs creates false new sessions
- Session gap too long (2 hours) → two distinct visits merged into one session
- Not handling state size → 2M users × unbounded state = crash
- Ignoring late data → mobile users with poor connectivity produce many late events

Add real-time session metrics to the observability dashboard (Task 119): sessions/minute, avg session duration, conversion rate.

   ```python
   class ChaosExperiment:
       def __init__(self, name: str, hypothesis: str):
           self.name = name
           self.hypothesis = hypothesis
           self.baseline_metrics = None

       def run(self):
           self.baseline_metrics = self.collect_metrics()  # Before
           print(f"[CHAOS] Starting: {self.name}")
           print(f"[CHAOS] Hypothesis: {self.hypothesis}")

           try:
               self.inject_failure()       # Break something
               self.wait_and_observe()     # Watch what happens
               result = self.verify()      # Check hypothesis
               self.report(result)
           finally:
               self.rollback()             # Always restore

       def collect_metrics(self) -> dict:
           return {
               'kafka_lag': get_consumer_lag(),
               'airflow_failures': get_failed_tasks(),
               'row_counts': get_table_row_counts(),
               'timestamp': datetime.utcnow(),
           }

       def verify(self) -> bool:
           after_metrics = self.collect_metrics()
           # Compare before/after — did recovery happen?
           return self.check_hypothesis(self.baseline_metrics, after_metrics)
   ```

### Common Mistakes
- Big-bang migration without reconciliation — load everything, hope it's correct
- Not deduplicating users — same person exists in both systems, now has two accounts
- Ignoring data quality issues — "we'll fix them later" — they compound
- No rollback plan — if migration corrupts NovaMart data, can you undo it?
- Not communicating to stakeholders — PM finds out about issues from a broken dashboard, not from you
