## Task 199: Retrospective — Lessons Learned Across 200 Tasks

### Business Context
Before the final capstone, take stock. You've completed nearly 200 tasks spanning junior and mid-level data engineering. Priya says: "The best engineers I know are deeply reflective. They don't just do the work — they think about what worked, what didn't, and what they'd teach their past selves."

### Objective
Write a comprehensive retrospective of your data engineering journey: technical lessons, leadership insights, and career advice for future mid-level engineers.

### Technical Requirements
- **Tools**: Markdown
- **Skills**: Self-reflection, knowledge synthesis, technical writing

### Step-by-Step Guidance

1. **Technical lessons**:
   ```markdown
   ## What I Learned About Data Engineering (200 Tasks Later)

   ### The Most Important Technical Lessons

   1. **Data quality is harder than data movement**
      Moving data from A to B is a solved problem. Ensuring it's correct, 
      complete, and timely is a career-long challenge. The dbt test that 
      catches a bug at 2 AM saves more value than the pipeline itself.

   2. **Streaming is not always better than batch**
      We spent weeks building a streaming pipeline that could have been 
      a 15-minute batch job. Real-time is only valuable when someone 
      acts on fresh data in real-time. Executive dashboards refreshed 
      hourly are fine for 90% of decisions.

   3. **Idempotency is non-negotiable**
      Every pipeline will be re-run. Every event will be duplicated. 
      If your pipeline isn't idempotent, you'll have bad data eventually. 
      This lesson cost me a weekend of incident response.

   4. **Schema changes are the #1 source of production incidents**
      Not hardware failures, not Kafka crashes — schema changes. 
      A column rename, a type change, a new nullable field — these 
      break more pipelines than any infrastructure failure.

   5. **Observability > monitoring**
      Monitoring says "alert when CPU > 90%." Observability says 
      "when the dashboard is stale, trace back to the exact failing 
      task in the exact failing DAG." Build for observability.

   ### Technical Regrets
   - Started with complex streaming too early — should have nailed batch first
   - Didn't implement data contracts until month 15 — should have been month 3
   - Over-complicated the first iteration of everything
   ```

2. **Leadership lessons**:
   ```markdown
   ### What I Learned About Leading a Technical Team

   1. **Hiring is the highest-leverage thing you do**
      Every hour spent finding the right person saves 100 hours of 
      coaching the wrong one. Our hiring process: take-home project → 
      system design → cultural fit. The take-home revealed more than 
      any whiteboard ever could.

   2. **1:1s are not status meetings**
      The best 1:1s I had with my reports covered career goals, 
      frustrations, and growth areas — not task updates. Task updates 
      belong in standups.

   3. **Documentation is a leadership tool**
      When I wrote clear ADRs, engineers stopped asking "why did we 
      choose X?" When I wrote good runbooks, on-call became less scary. 
      When I wrote the operations manual, I could take vacation.

   4. **Saying "no" is a skill**
      Every domain team wants custom work. Saying "no, but here's 
      how you can do it yourself using our self-serve tools" is 
      the path to a scalable platform.

   5. **The bus factor is your responsibility**
      If only you understand the system, that's a failure of leadership, 
      not a testament to your indispensability.
   ```

3. **Advice for future mid-level DEs**:
   ```markdown
   ### Advice I'd Give My Past Self (and Future Mid-Level DEs)

   1. **Learn to write before you learn to code**
      The ADR, the postmortem, the strategy doc, the blog post — 
      writing clearly is the multiplier that makes everything else 
      more impactful. Code is read more than it's written; 
      documents are too.

   2. **Measure everything, but only report what matters**
      Track 50 metrics internally. Report 5 to your manager. 
      Report 2 to the VP. Know which 2 those are.

   3. **The promotion happens before the promotion**
      I didn't get promoted to Senior and then start doing 
      Senior work. I started doing Senior work (strategy docs, 
      cross-team influence, mentoring) and the promotion followed.

   4. **Build relationships before you need them**
      When the CEO calls about the broken dashboard at 11:40 AM, 
      you need Ravi to trust you, Priya to back you, and the 
      domain team to believe your fix. Build that trust when 
      things are calm.

   5. **Take care of yourself**
      I burned out twice during this journey. The second time, 
      I learned: sustainable pace > heroic pace. A rested engineer 
      makes better decisions than a tired one, every time.
   ```

### Expected Output
- Full retrospective: `tasks/199_retrospective/retrospective.md`
- Technical lessons learned
- Leadership lessons learned
- Advice document

### Common Mistakes
- Being too abstract — specific examples are more memorable and useful than generalizations
- Only highlighting successes — the failures teach more; include them
- Not being honest about regrets — "everything went great" is not realistic or helpful
- Writing only for yourself — format it so a junior DE joining next month finds it valuable

### Stretch Goal
Share your retrospective internally (engineering all-hands or blog). Vulnerability and openness from a senior-track engineer inspires the rest of the team.

## Task 200: Final Mid-Level Capstone — The Complete Data Platform

### Business Context
This is it. Your final task as a Mid-Level Data Engineer. Priya and Ravi have set up a comprehensive capstone challenge: design, document, and present a complete data platform architecture from scratch, as if NovaMart were starting over today with everything you've learned. This is the definitive test of mid-level mastery and senior readiness.

### Objective
Design a complete, production-ready data platform architecture for a multi-region e-commerce company, incorporating every major lesson from Tasks 1-199. This is your magnum opus.

### The Challenge
> "A new e-commerce company called StellarShop (competitor to NovaMart) has raised $50M Series B funding. They operate in 5 countries, process 50K orders/day, and have 15 engineers. They have NO data platform — everything is in production PostgreSQL. Design their data platform from scratch. You have a $100K/month budget and need to hire a team."

### Technical Requirements
- **Deliverables**: Architecture document, tech stack justification, team plan, phased roadmap, cost model
- **Time**: This is a significant project; expect 20-30 hours of work
- **Evaluation**: Priya will review for technical depth; Ravi for pragmatism

### Step-by-Step Guidance

1. **Requirements gathering** (document assumptions):
   ```markdown
   ## StellarShop Data Platform — Design Document

   ### Company Context
   - 5 countries (US, UK, Canada, Australia, Japan)
   - 50K orders/day (~2K/hour, ~35/minute peak)
   - 15 engineers total (3 dedicated to data)
   - $100K/month infrastructure budget
   - PostgreSQL production DB, no data warehouse
   - Goals: analytics dashboards, ML recommendations, regulatory compliance

   ### Functional Requirements
   1. Executive dashboards (revenue, orders, inventory) — daily refresh minimum
   2. Self-serve analytics for business teams (Marketing, Finance, Ops)
   3. ML feature serving for product recommendations
   4. GDPR/CCPA compliance for customer data
   5. Multi-region data sovereignty (Japan data stays in APAC)

   ### Non-Functional Requirements
   1. Data freshness: < 1 hour for critical metrics, < 4 hours for all
   2. Availability: 99.5% SLA for dashboards
   3. Cost efficiency: < $2K per pipeline per month
   4. Scalability: handle 5x growth in 12 months
   5. Team of 3 DEs must be able to operate it (no 10-person team required)
   ```

2. **Architecture design**:
   ```markdown
   ### High-Level Architecture

   ```mermaid
   graph TB
       subgraph Sources
           PG[PostgreSQL Production]
           APP[App Events - Mobile/Web]
           THIRD[Third Party - Payment/Shipping]
       end

       subgraph Ingestion
           CDC[Debezium CDC]
           KAFKA[Kafka / Confluent Cloud]
           FIVETRAN[Fivetran - Third Party]
       end

       subgraph Storage
           S3[S3 Data Lake - Iceberg]
           SF[Snowflake - Warehouse]
       end

       subgraph Transform
           DBT[dbt Core]
           AIRFLOW[Airflow / MWAA]
       end

       subgraph Serve
           LOOKER[Looker - Dashboards]
           API[Data API - FastAPI]
           REDIS[Redis - Feature Serving]
       end

       subgraph Ops
           GH[GitHub Actions - CI/CD]
           TF[Terraform - IaC]
           GRAF[Grafana - Monitoring]
       end

       PG --> CDC --> KAFKA
       APP --> KAFKA
       THIRD --> FIVETRAN
       KAFKA --> S3
       FIVETRAN --> SF
       S3 --> SF
       SF --> DBT
       DBT --> SF
       SF --> LOOKER
       SF --> API --> REDIS
       AIRFLOW --> DBT
       GH --> AIRFLOW
       TF --> SF
       GRAF --> KAFKA
   ```

   ### Tech Stack Justification

   | Component | Choice | Why | Alternative Considered |
   |-----------|--------|-----|----------------------|
   | Warehouse | Snowflake | Managed, scalable, multi-region | BigQuery (less SQL-compatible) |
   | Orchestration | Airflow (MWAA) | Managed, Python-native, extensible | Dagster (less mature ecosystem) |
   | Transform | dbt Core | SQL-native, version controlled, tested | Custom Python (slower to develop) |
   | Streaming | Confluent Cloud | Managed Kafka, schema registry included | Self-managed Kafka (too much ops) |
   | CDC | Debezium | Open-source, Postgres native | Fivetran (more expensive for CDC) |
   | Data Lake | S3 + Iceberg | Open format, avoids lock-in | Delta Lake (Databricks-centric) |
   | BI | Looker | SQL-native, self-serve, governance | Metabase (less scalable) |
   | IaC | Terraform | Multi-cloud, declarative, mature | Pulumi (smaller community) |
   | CI/CD | GitHub Actions | Integrated with repo, simple | Jenkins (over-engineered for 3 DEs) |
   | Monitoring | Grafana + PagerDuty | Open-source, flexible | Datadog (expensive for startup) |

   **Key principle**: Managed services over self-hosted. With 3 DEs,
