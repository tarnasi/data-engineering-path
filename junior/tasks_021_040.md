## Task 21: SQL Query Optimization — The Slow Dashboard

### Business Context
The BI team's main dashboard takes 45 seconds to load. The CEO opens it every morning and complains. Priya forwards you the ticket: "The underlying query is too slow. Optimize it." This is your first performance tuning task.

### Objective
Analyze a slow SQL query using `EXPLAIN ANALYZE`, identify bottlenecks, and optimize it using indexing, query restructuring, and materialized views.

### Technical Requirements
- **Tools**: SQL (PostgreSQL), EXPLAIN ANALYZE
- **Skills**: Query plans, sequential scan vs index scan, index creation, materialized views, query rewriting

### Dataset
NovaMart database with all tables from previous tasks. Ensure `orders` has 100K+ rows for meaningful performance testing. Use the Olist dataset or generate synthetic data:
```sql
-- Generate 200K orders for performance testing
INSERT INTO orders (user_id, order_date, status, total_amount, payment_method)
SELECT
    (random() * 10000 + 1)::int,
    '2023-01-01'::date + (random() * 730)::int * interval '1 day',
    (ARRAY['pending','confirmed','shipped','delivered','cancelled','returned'])[floor(random()*6+1)::int],
    (random() * 500 + 5)::numeric(10,2),
    (ARRAY['credit_card','debit_card','paypal','apple_pay'])[floor(random()*4+1)::int]
FROM generate_series(1, 200000);
```

### Step-by-Step Guidance
1. The slow query (simulating the dashboard):
   ```sql
   SELECT
       o.order_date::date AS day,
       COUNT(DISTINCT o.order_id) AS orders,
       SUM(p.amount) AS revenue,
       COUNT(DISTINCT o.user_id) AS unique_buyers,
       AVG(p.amount) AS avg_payment
   FROM orders o
   JOIN payments p ON o.order_id = p.order_id
   JOIN order_items oi ON o.order_id = oi.order_id
   JOIN products pr ON oi.product_id = pr.product_id
   WHERE o.order_date >= NOW() - INTERVAL '90 days'
     AND o.status IN ('confirmed', 'shipped', 'delivered')
     AND p.status = 'success'
   GROUP BY o.order_date::date
   ORDER BY day DESC;
   ```
2. Run `EXPLAIN ANALYZE` on the query — save the output
3. Identify: which joins cause sequential scans? Is there a missing index?
4. Create targeted indexes:
   ```sql
   CREATE INDEX idx_orders_date_status ON orders(order_date, status);
   CREATE INDEX idx_payments_order_status ON payments(order_id, status);
   ```
5. Re-run `EXPLAIN ANALYZE` — compare before/after execution time
6. If still slow, create a materialized view:
   ```sql
   CREATE MATERIALIZED VIEW mv_daily_dashboard AS (/* optimized query */);
   ```
7. Document: original time, optimized time, what changes helped, trade-offs

### Expected Output
- Analysis doc: `tasks/21_query_optimization/analysis.md` (before/after EXPLAIN plans)
- Index DDL: `tasks/21_query_optimization/indexes.sql`
- Materialized view: `tasks/21_query_optimization/materialized_view.sql`
- Performance improvement: target >5x speedup

### Common Mistakes
- Adding indexes on every column "just in case" — indexes slow down writes and use disk space
- Not running `ANALYZE` on tables after bulk loads (statistics are stale, planner makes bad decisions)
- Using `SELECT *` when you only need 3 columns — forces full row reads
- Creating a materialized view but forgetting to schedule `REFRESH MATERIALIZED VIEW`

### Stretch Goal
Set up a scheduled `REFRESH MATERIALIZED VIEW CONCURRENTLY` in Airflow (non-blocking refresh).

## Task 25: Data Deduplication at Scale

### Business Context
After NovaMart's acquisition (Task 5), duplicate customer records are everywhere. User #1234 on the old system and User #5678 on NovaMart might be the same person (same email, different user_id). The customer support team reports issues daily. Priya assigns: "Build a deduplication pipeline."

### Objective
Write a deduplication pipeline that identifies and merges duplicate user records using exact matching and fuzzy matching techniques.

### Technical Requirements
- **Tools**: Python (pandas, fuzzywuzzy/rapidfuzz), SQL
- **Skills**: Exact dedup, fuzzy matching, record linkage, merge strategies, golden record creation

### Dataset
Create a merged user table with intentional duplicates:
```csv
source_system,user_id,full_name,email,phone,country,created_at
novamart,1,Alice Smith,alice@example.com,555-0101,US,2023-01-15
novamart,2,Bob Jones,bob.jones@example.com,555-0202,US,2023-02-20
acquired,101,alice smith,alice@example.com,,US,2022-06-01
acquired,102,Bobby Jones,bob.jones@example.com,555-0202,CA,2022-08-15
novamart,3,Charlie Brown,charlie.b@example.com,555-0303,US,2023-03-10
acquired,103,Charles Brown,charlieb@example.com,555-0303,US,2022-05-20
novamart,4,Diana Prince,diana@example.com,,US,2023-04-01
acquired,104,Diana Prince,d.prince@example.com,555-0404,US,2022-07-10
```

### Step-by-Step Guidance
1. **Exact match**: Group by normalized email → these are definite duplicates
2. **Fuzzy match**: For remaining records, compare:
   - Name similarity (Levenshtein distance > 85%)
   - Phone number match
   - Same country + similar name
3. **Score each pair**: exact email = 100, same phone = 80, fuzzy name > 85% = 60, combine scores
4. **Create golden records**: For each duplicate group, choose the "best" record:
   - Prefer NovaMart's `user_id` as canonical
   - Take the earliest `created_at`
   - Fill missing fields from the duplicate (e.g., phone from acquired system)
5. **Output**:
   - `duplicate_pairs.csv`: pairs of user_ids identified as duplicates with confidence score
   - `golden_records.csv`: merged/canonical records
   - `mapping_table.csv`: old_user_id → canonical_user_id

### Expected Output
- Pipeline: `tasks/25_deduplication/dedup_users.py`
- Output files in `tasks/25_deduplication/output/`
- Summary: X users input, Y duplicate groups found, Z golden records produced

### Common Mistakes
- Only matching on one field (email) — misses duplicates with typos or different email addresses
- Fuzzy matching everything against everything (O(n²)) — use blocking (group by country or first letter) first
- Not handling transitive duplicates: if A=B and B=C, then A=C (need union-find/clustering)
- Overwriting good data with the duplicate's worse data during merge

### Stretch Goal
Build a review queue: for pairs with confidence 60-80%, output them for human review instead of auto-merging.

Before building anything, Priya wants a Technical Design Document (TDD). "Senior engineers write design docs before writing code. Show me you can think through a system before building it."

### Objective
Write a complete Technical Design Document for the Canada expansion data pipeline — covering requirements, architecture, data model changes, pipeline design, edge cases, and rollout plan.

### Technical Requirements
- **Tools**: Markdown, diagramming tool (draw.io, Mermaid, or ASCII)
- **Skills**: Technical writing, system design thinking, trade-off analysis, requirement decomposition

### Dataset
No coding required — this is a design exercise. Reference all NovaMart tables and pipelines from Tasks 1-39.

### Step-by-Step Guidance
Write a document with these sections:

1. **Overview**: What are we building and why?
2. **Requirements**:
   - Functional: what the pipeline must do
   - Non-functional: latency, reliability, data freshness SLAs
3. **Data Model Changes**:
   - New columns needed (currency, tax_province, language)
   - Changes to dim_users, dim_products, fact_order_items
   - How to handle multi-currency in the star schema
4. **Architecture Diagram**:
   - Show data flow from Canadian sources through to dashboards
   - Include exchange rate API, tax calculation service
5. **Pipeline Design**:
   - New DAG or modify existing?
   - How to handle currency conversion (at order time vs at reporting time)
   - Tax calculation: lookup table vs API call
6. **Edge Cases**:
   - User in Quebec (French + unique tax rules)
   - Cross-border orders (US user shipping to Canada)
   - Exchange rate fluctuations mid-day
7. **Rollout Plan**:
   - Phase 1: Schema changes (backwards compatible)
   - Phase 2: Pipeline updates
   - Phase 3: Dashboard updates
   - Rollback plan if something goes wrong
8. **Open Questions**: List uncertainties that need stakeholder input

### Expected Output
- Design doc: `tasks/40_design_doc/canada_expansion_tdd.md`
- Architecture diagram (Mermaid or image)
- At least 3 pages of substantive technical content

### Common Mistakes
- Jumping to implementation without listing requirements first
- Not considering backwards compatibility — existing US pipelines must keep working
- Ignoring edge cases — the Canada pipeline will get edge cases on day 1
- No rollback plan — "what if this breaks production?" needs an answer
- Making the document too high-level ("we'll use Airflow") without enough detail to implement from

### Stretch Goal
Include a cost estimate: how many additional Airflow tasks, additional DB storage, API call costs for exchange rates, estimated pipeline runtime impact.
