## Task 62: Advanced SQL — Recursive Queries and Graph Traversal

### Business Context
NovaMart introduced a referral program. Users refer friends, who refer more friends, forming a referral chain. The marketing team wants to know: "For each user, how deep is their referral chain? Who referred the most total downstream users (direct + indirect)?" This requires recursive SQL.

### Objective
Write recursive CTEs to traverse the referral graph, calculate referral chain depth, and find the most influential referrers.

### Technical Requirements
- **Tools**: SQL (PostgreSQL recursive CTEs)
- **Skills**: Recursive CTEs, graph traversal, cycle detection, hierarchical data

### Dataset
Create a referral table:
```sql
CREATE TABLE referrals (
    user_id INTEGER PRIMARY KEY,
    referred_by INTEGER REFERENCES referrals(user_id),
    referral_date DATE,
    signup_date DATE
);

INSERT INTO referrals VALUES
  (1, NULL, NULL, '2023-01-01'),          -- organic signup (root)
  (2, 1, '2023-02-01', '2023-02-01'),     -- referred by user 1
  (3, 1, '2023-02-15', '2023-02-15'),     -- referred by user 1
  (4, 2, '2023-03-01', '2023-03-01'),     -- referred by user 2
  (5, 2, '2023-03-10', '2023-03-10'),     -- referred by user 2
  (6, 4, '2023-04-01', '2023-04-01'),     -- referred by user 4
  (7, 4, '2023-04-15', '2023-04-15'),     -- referred by user 4
  (8, 6, '2023-05-01', '2023-05-01'),     -- referred by user 6
  (9, 3, '2023-03-20', '2023-03-20'),     -- referred by user 3
  (10, NULL, NULL, '2023-06-01');          -- organic signup (another root)
```

### Step-by-Step Guidance
1. **Find referral chain for any user** (walk up to the root):
   ```sql
   WITH RECURSIVE chain AS (
       SELECT user_id, referred_by, 0 AS depth
       FROM referrals
       WHERE user_id = 8  -- starting user

       UNION ALL

       SELECT r.user_id, r.referred_by, c.depth + 1
       FROM referrals r
       JOIN chain c ON r.user_id = c.referred_by
   )
   SELECT * FROM chain;
   ```
2. **Calculate each user's total downstream referrals** (walk down from each user):
   ```sql
   WITH RECURSIVE downstream AS (
       SELECT referred_by AS referrer_id, user_id AS downstream_user, 1 AS depth
       FROM referrals
       WHERE referred_by IS NOT NULL

       UNION ALL

       SELECT d.referrer_id, r.user_id, d.depth + 1
       FROM downstream d
       JOIN referrals r ON r.referred_by = d.downstream_user
   )
   SELECT referrer_id, COUNT(DISTINCT downstream_user) AS total_downstream, MAX(depth) AS max_chain_depth
   FROM downstream
   GROUP BY referrer_id
   ORDER BY total_downstream DESC;
   ```
3. **Add cycle detection** (prevent infinite loops if data has circular references):
   ```sql
   WITH RECURSIVE chain AS (
       SELECT user_id, referred_by, ARRAY[user_id] AS visited, FALSE AS has_cycle
       FROM referrals WHERE user_id = 1
       UNION ALL
       SELECT r.user_id, r.referred_by, c.visited || r.user_id,
              r.user_id = ANY(c.visited) AS has_cycle
       FROM referrals r
       JOIN chain c ON r.referred_by = c.user_id
       WHERE NOT c.has_cycle
   )
   SELECT * FROM chain;
   ```
4. **Find longest referral chain in the entire system**
5. **Revenue attribution**: for each referrer, sum the revenue of all downstream users

### Expected Output
- SQL script: `tasks/62_recursive_sql/referral_queries.sql`
- Output showing: top referrers, chain depths, revenue attribution
- Cycle detection demonstration

### Common Mistakes
- No recursion depth limit — infinite loops crash the database (`LIMIT` or cycle detection required)
- Confusing "walking up" (to root) vs "walking down" (to leaves) — the JOIN direction flips
- Not handling multiple roots (users with `referred_by = NULL`)
- Counting direct referrals only, missing the recursive downstream count

```csv
S001,WH-East,temperature,22.5,2024-07-15 00:00:00
S001,WH-East,temperature,22.8,2024-07-15 00:05:00
S001,WH-East,temperature,23.1,2024-07-15 00:10:00
S001,WH-East,temperature,,2024-07-15 00:15:00
```

### Step-by-Step Guidance

   ```sql

       sensor_id VARCHAR(20),
       warehouse_id VARCHAR(20),
       metric VARCHAR(50),
       value NUMERIC(10,2),
       timestamp TIMESTAMP,
       is_interpolated BOOLEAN DEFAULT FALSE,
       PRIMARY KEY (sensor_id, metric, timestamp)
   );
   ```
2. **Gap detection**: identify missing 5-minute intervals
   ```python
   expected_times = pd.date_range(start=df['timestamp'].min(), end=df['timestamp'].max(), freq='5min')
   missing = expected_times.difference(df['timestamp'])
   ```
3. **Gap filling**: interpolate missing values (linear interpolation for temperature)
4. **Downsample**: create hourly averages from 5-minute readings
5. **Anomaly detection**: flag readings that are >3 standard deviations from the rolling 1-hour mean
   ```python
   df['rolling_mean'] = df.groupby(['sensor_id', 'metric'])['value'].transform(
       lambda x: x.rolling(window=12, min_periods=6).mean()  # 12 x 5min = 1 hour
   )
   df['rolling_std'] = df.groupby(['sensor_id', 'metric'])['value'].transform(
       lambda x: x.rolling(window=12, min_periods=6).std()
   )
   df['is_anomaly'] = abs(df['value'] - df['rolling_mean']) > 3 * df['rolling_std']
   ```
6. **Alerting**: if any sensor shows temperature > 30°C or anomaly detected, log an alert

- Alert log

### Common Mistakes
- Forward-filling gaps across large time periods — a 24-hour gap shouldn't be interpolated
- Not handling per-sensor gap detection — each sensor has its own expected schedule
- Downsampling with `mean()` and losing min/max — store min, max, avg, and count for each hour
- Setting anomaly thresholds without understanding the data distribution first

## Task 67: Spark Joins and Broadcast Optimization

### Business Context
Your Spark pipeline joins a 500M-row clickstream table with a 10K-row product dimension. The join takes 20 minutes because Spark shuffles both tables. The platform engineer looks at your code and says: "Broadcast the small table. This should take 2 minutes, not 20."

### Technical Requirements
- **Tools**: PySpark, Spark UI
- **Skills**: Join strategies, broadcast joins, skew handling, join optimization

### Dataset
- Large: clickstream Parquet (5M+ rows)
- Small: products dimension (10K rows), users dimension (50K rows)
- Skewed: orders with a "guest" user_id that accounts for 30% of rows

### Step-by-Step Guidance
1. **Default join** (sort-merge, shuffles both sides):
   ```python
   result = df_clickstream.join(df_products, "product_id")
   result.explain()  # Show the plan — look for Exchange (shuffle)
   ```
2. **Broadcast join** (small table broadcast to all executors):
   ```python
   from pyspark.sql.functions import broadcast

   ```

4. **Handle join skew** (the "guest" user problem):
   ```python
   # Option A: Salt the skewed key
   import pyspark.sql.functions as F

   df_orders_salted = df_orders.withColumn(
       "salt", F.when(F.col("user_id") == 0, F.floor(F.rand() * 10)).otherwise(0)
   ).withColumn("user_id_salted", F.concat(F.col("user_id"), F.lit("_"), F.col("salt")))

   # Option B: Separate the skewed key, process it differently, union back
   df_guest = df_orders.filter(F.col("user_id") == 0)
   df_regular = df_orders.filter(F.col("user_id") != 0)
   ```

   ```python
   spark.conf.set("spark.sql.autoBroadcastJoinThreshold", 50 * 1024 * 1024)  # 50MB
   ```
6. Document: when to use each strategy, with size thresholds

### Expected Output
- Join comparison script: `tasks/67_spark_joins/join_strategies.py`
- Performance comparison: default vs broadcast vs salted join
- Spark UI screenshots or explain plan outputs
- Guide: which join strategy for which scenario

### Common Mistakes
- Broadcasting a large table — runs out of driver memory and crashes
- Not checking if auto-broadcast is already happening (Spark may broadcast small tables by default)

## Task 68: Build a Self-Service Data Request System

### Business Context

Most are simple queries that analysts could run themselves if they had the right tools. Priya says: "Build a self-service layer so analysts can answer their own questions."

### Objective
Build a set of parameterized query templates that analysts can run via a simple interface, without needing to write SQL from scratch.

### Technical Requirements
- **Tools**: Python (Jinja2 for templating), SQL, Streamlit or CLI
- **Skills**: Query templating, parameterization, safe query execution, audit logging

### Dataset
NovaMart analytical tables (star schema, customer segments, daily summary).

### Step-by-Step Guidance
1. Create query templates:
   ```yaml
   # templates/user_purchase_list.yaml
   name: "User Purchase List"
   description: "Get list of users who purchased a specific product category in a date range"
   parameters:
     - name: category
       type: string
       required: true
       allowed_values: [electronics, fashion, home, sports]
     - name: start_date
       type: date
       required: true
     - name: end_date
       type: date
       required: true
     - name: min_spend
       type: number
       required: false
       default: 0
   query: |
     SELECT u.user_id, u.email, p.category,
            SUM(oi.unit_price * oi.quantity) AS total_spend,
            COUNT(DISTINCT o.order_id) AS order_count
     FROM fact_order_items oi
     JOIN dim_users u ON oi.user_key = u.user_key AND u.is_current = TRUE
     JOIN dim_products p ON oi.product_key = p.product_key
     JOIN dim_date d ON oi.date_key = d.date_key
     WHERE p.category = :category
       AND d.date BETWEEN :start_date AND :end_date
       AND (oi.unit_price * oi.quantity) >= :min_spend
     GROUP BY u.user_id, u.email, p.category
     ORDER BY total_spend DESC
   ```
2. Build a template executor:

### Business Context
The ML team built a churn prediction model that needs a daily feature table. They defined the feature requirements in a Google Doc six months ago.          - name: user_id

   ```yaml
         - name: feature_date

           required: true
         - name: days_since_last_order

           type: decimal
           required: false
           deprecated: false

       - version: 2.1
   ```

   ```python
   class InterfaceValidator:

           spec = load_spec(spec_path)
           actual_schema = get_table_schema(table)
           # Check: all required columns present, types match, valid ranges respected
           # Check: no unexpected columns added (warn, don't fail)

       def validate_breaking_change(self, old_spec: str, new_spec: str) -> List[BreakingChange]:
   ```

## Task 78: Build a Pipeline Cost Estimation Tool

### Business Context
NovaMart's cloud bill for data infrastructure tripled last month. The CFO wants to know: "How much does each pipeline cost to run?" Currently, nobody knows. Priya says: "Build a cost model. We need to know cost per pipeline, cost per table, and cost trends."

### Objective
Build a tool that estimates the compute, storage, and I/O cost of each data pipeline based on resource usage metrics.

### Technical Requirements
- **Tools**: Python, SQL, Airflow metadata
- **Skills**: Cost modeling, resource estimation, cloud pricing models, efficiency metrics

### Dataset
Pipeline run metadata, table sizes, query execution stats.

### Step-by-Step Guidance
1. Collect resource usage data:
   ```sql
   -- Table sizes
   SELECT
       relname AS table_name,
       pg_size_pretty(pg_total_relation_size(relid)) AS total_size,
       pg_total_relation_size(relid) AS size_bytes,
       n_live_tup AS row_count
   FROM pg_stat_user_tables
   ORDER BY pg_total_relation_size(relid) DESC;

   -- Query execution stats (if pg_stat_statements enabled)
   SELECT
       query,
       calls,
       total_exec_time / 1000 AS total_seconds,
       rows,
       shared_blks_read + shared_blks_hit AS total_blocks
   FROM pg_stat_statements
   ORDER BY total_exec_time DESC;
   ```
2. Define a cost model:
   ```python
   class CostModel:
       # Simulated cloud pricing
       COMPUTE_COST_PER_HOUR = 0.50      # per vCPU-hour
       STORAGE_COST_PER_GB_MONTH = 0.023  # S3-like pricing
       IO_COST_PER_MILLION_ROWS = 0.005   # data scanned

       def estimate_pipeline_cost(self, pipeline_name: str) -> dict:
           runs = get_pipeline_runs(pipeline_name, days=30)
           compute = sum(r.duration_hours * self.COMPUTE_COST_PER_HOUR for r in runs)
           storage = get_output_table_size_gb(pipeline_name) * self.STORAGE_COST_PER_GB_MONTH
           io = sum(r.rows_processed for r in runs) / 1_000_000 * self.IO_COST_PER_MILLION_ROWS
           return {'compute': compute, 'storage': storage, 'io': io, 'total': compute + storage + io}
   ```
3. Generate a cost report:
   | Pipeline | Monthly Compute | Storage | I/O | Total | Cost/Run |
   |----------|----------------|---------|-----|-------|----------|
   | clickstream_ingest | $15.00 | $8.50 | $3.20 | $26.70 | $0.89 |
   | dbt_marts | $5.00 | $2.30 | $1.10 | $8.40 | $0.28 |
4. Identify optimization opportunities:
   - Which pipeline has the highest cost per row?
   - Which tables are large but rarely queried? (candidates for archiving)
   - Which pipelines run too frequently for their SLA?
5. Trend analysis: is cost growing linearly with data volume, or faster?

### Expected Output
- Cost model: `tasks/78_cost_estimation/cost_model.py`
- Resource collector: `tasks/78_cost_estimation/resource_collector.py`
- Cost report: `tasks/78_cost_estimation/cost_report.md`
- At least 3 optimization recommendations

### Common Mistakes
- Only counting compute, ignoring storage — storage compounds monthly
- Not measuring cost per row — high total cost on a pipeline processing 100M rows may be efficient
- Using list prices instead of actual usage — reserved instances, spot pricing, committed use discounts change the math
- Point-in-time snapshot instead of trends — last month's cost alone doesn't show growth trajectory

### Stretch Goal
Build a cost dashboard (Streamlit) showing per-pipeline costs, trends, and "waste" (tables that haven't been queried in 30+ days).
