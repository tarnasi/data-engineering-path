## Task 82: Implement a Dead Letter Queue for Failed Records

### Business Context
The clickstream pipeline processes 10M events/day. About 0.1% fail — malformed JSON, missing fields, encoding errors. Currently, the pipeline either crashes on the first bad record or silently drops bad records. Neither is acceptable. Priya says: "Good records should flow through. Bad records should be captured, not lost."

### Objective
Implement a dead letter queue (DLQ) pattern: good records proceed through the pipeline, bad records are routed to a separate storage location with error metadata, available for investigation and reprocessing.

### Technical Requirements
- **Tools**: Python, SQL, file storage (for DLQ)
- **Skills**: Error isolation, dead letter queues, record-level error handling, reprocessing

### Dataset
Clickstream events (10,000+ records), with ~1% intentionally malformed.

### Step-by-Step Guidance
1. Define the DLQ schema:
   ```sql
   CREATE TABLE dead_letter_queue (
       dlq_id SERIAL PRIMARY KEY,
       source_pipeline VARCHAR(100),
       source_file VARCHAR(500),
       record_number INTEGER,
       raw_record TEXT,
       error_type VARCHAR(100),
       error_message TEXT,
       error_timestamp TIMESTAMP DEFAULT NOW(),
       reprocessed BOOLEAN DEFAULT FALSE,
       reprocessed_at TIMESTAMP
   );
   ```

2. Build the error-isolating processor:
   ```python
   class ResilientProcessor:
       def __init__(self, pipeline_name: str):
           self.pipeline_name = pipeline_name
           self.good_records = []
           self.dead_letters = []

       def process_batch(self, records: List[dict]) -> ProcessResult:
           for i, record in enumerate(records):
               try:
                   validated = self.validate(record)
                   transformed = self.transform(validated)
                   self.good_records.append(transformed)
               except ValidationError as e:
                   self.dead_letters.append(DeadLetter(
                       record=record,
                       error_type='validation',
                       error_message=str(e),
                       record_number=i
                   ))
               except TransformError as e:
                   self.dead_letters.append(DeadLetter(
                       record=record,
                       error_type='transform',
                       error_message=str(e),
                       record_number=i
                   ))

           return ProcessResult(
               good_count=len(self.good_records),
               dead_count=len(self.dead_letters),
               dead_letter_rate=len(self.dead_letters) / len(records)
           )
   ```

3. Write dead letters to both database and file:
   ```python
   def flush_dead_letters(self):
       # Write to database for querying
       insert_to_dlq_table(self.dead_letters)
       # Write raw records to file for bulk reprocessing
       write_jsonl(f"dlq/{self.pipeline_name}/{date}/dead_letters.jsonl", self.dead_letters)
   ```

4. Build a reprocessing mechanism:
   ```python
   class DLQReprocessor:
       def reprocess(self, pipeline_name: str, date: str, fix_function=None):
           """Re-run dead letters through the pipeline, optionally with a fix applied"""
           dead_letters = load_dead_letters(pipeline_name, date)
           for dl in dead_letters:
               record = dl.raw_record
               if fix_function:
                   record = fix_function(record)
               try:
                   process_single(record)
                   mark_reprocessed(dl.dlq_id)
               except Exception as e:
                   update_error(dl.dlq_id, str(e))  # still failing
   ```

5. Add alerting: if DLQ rate exceeds 1%, alert — something systematic is wrong
6. DLQ dashboard: count by error type, source, date — identify patterns

### Expected Output
- Processor: `tasks/82_dead_letter_queue/resilient_processor.py`
- Reprocessor: `tasks/82_dead_letter_queue/reprocessor.py`
- DLQ table creation script
- Demo: process 10,000 records with 100 bad ones → 9,900 loaded, 100 in DLQ, alert generated

### Common Mistakes
- Catching all exceptions and swallowing them — you must log the error detail, not just skip the record
- DLQ grows forever — set retention policies, archive after 30 days
- Not preserving the raw record — you transformed it before catching the error, now you can't replay the original
- Reprocessing without fixing the root cause — the same records fail again

### Stretch Goal
Build a DLQ analysis tool that clusters dead letters by error type and suggests common fixes (e.g., "73% of failures are encoding errors on the `review_text` field").

## Task 83: Advanced Python — Generators, Context Managers, and Memory Efficiency

### Business Context
Your vendor inventory pipeline loads a 2GB CSV entirely into memory as a DataFrame. The server has 4GB RAM. It works fine until the file grows to 5GB — then it crashes with `MemoryError`. Priya says: "You need to learn streaming patterns. Process data in chunks, not all at once."

### Objective
Refactor a memory-heavy pipeline using Python generators, chunked processing, and context managers to handle datasets larger than available RAM.

### Technical Requirements
- **Tools**: Python (generators, itertools, contextlib)
- **Skills**: Memory-efficient processing, generators, context managers, chunked I/O

### Dataset
Generate a large CSV (1M+ rows, ~500MB) representing vendor inventory data.

### Step-by-Step Guidance
1. **The problem** — loading everything into memory:
   ```python
   # BAD: loads entire file into memory
   df = pd.read_csv("large_vendor_file.csv")  # 2GB in RAM
   processed = df[df['quantity'] > 0]           # 2GB copy
   processed['margin'] = processed['price'] - processed['cost']  # another copy
   # Peak memory: ~6GB for a 2GB file
   ```

2. **Solution 1: pandas chunked reading**:
   ```python
   def process_chunked(filepath: str, chunk_size: int = 50_000):
       chunks = pd.read_csv(filepath, chunksize=chunk_size)
       for chunk in chunks:
           filtered = chunk[chunk['quantity'] > 0]
           filtered['margin'] = filtered['price'] - filtered['cost']
           yield filtered
   ```

3. **Solution 2: Pure Python generator for CSV**:
   ```python
   import csv
   from typing import Generator, Dict

   def read_csv_streaming(filepath: str) -> Generator[Dict, None, None]:
       with open(filepath, 'r') as f:
           reader = csv.DictReader(f)
           for row in reader:
               yield row  # one row at a time, constant memory

   def filter_and_transform(rows: Generator) -> Generator:
       for row in rows:
           if int(row['quantity']) > 0:
               row['margin'] = float(row['price']) - float(row['cost'])
               yield row
   ```

4. **Build a streaming pipeline with generators**:
   ```python
   def streaming_pipeline(filepath: str, output_path: str, batch_size: int = 10_000):
       rows = read_csv_streaming(filepath)
       transformed = filter_and_transform(rows)
       batched = batch_iterator(transformed, batch_size)

       for batch in batched:
           write_to_database(batch)  # insert 10K rows at a time
   ```

5. **Context managers for resource management**:
   ```python
   from contextlib import contextmanager

   @contextmanager
   def pipeline_connection(config: dict):
       conn = psycopg2.connect(**config)
       try:
           yield conn
           conn.commit()
       except Exception:
           conn.rollback()
           raise
       finally:
           conn.close()

   # Usage
   with pipeline_connection(db_config) as conn:
       for batch in batched:
           write_batch(conn, batch)
   ```

6. **Measure memory**: use `tracemalloc` to compare before/after
   ```python
   import tracemalloc
   tracemalloc.start()
   # ... run pipeline ...
   current, peak = tracemalloc.get_traced_memory()
   print(f"Peak memory: {peak / 1024 / 1024:.1f} MB")
   ```

7. Compare: full-load vs chunked vs streaming — memory usage and execution time

### Expected Output
- Full-load version: `tasks/83_memory_efficiency/pipeline_naive.py`
- Streaming version: `tasks/83_memory_efficiency/pipeline_streaming.py`
- Memory comparison report showing peak memory reduction (e.g., 2GB → 50MB)
- Benchmark: execution time comparison

### Common Mistakes
- Generators consumed twice — a generator is single-use; calling `list()` on it loads everything into memory (defeating the purpose)
- Not batching database writes — inserting row by row is 100x slower than batching
- Chunked pandas still copies on transform — use `inplace=True` or pre-allocate
- Forgetting to close files/connections — context managers prevent this

### Stretch Goal
Implement a `pipeline_memory_profiler` decorator that wraps any function, tracks peak memory, and logs it to the monitoring table.

## Task 84: Build a Schema Registry

### Business Context
NovaMart's data sources change their schemas without warning. Last week, the clickstream API added a `device_type` field. The week before, the vendor system renamed `sku` to `product_sku`. Each time, a pipeline breaks. Priya says: "We need a schema registry — a central place that defines expected schemas, detects changes, and alerts before they cause damage."

### Objective
Build a schema registry that stores expected schemas for all data sources, compares incoming data against the registry, and handles schema evolution gracefully.

### Technical Requirements
- **Tools**: Python, JSON Schema or custom validation, versioned storage
- **Skills**: Schema management, schema evolution, compatibility checking, versioning

### Dataset
Schemas for all NovaMart data sources (clickstream events, vendor files, API payloads).

### Step-by-Step Guidance
1. Define schema format:
   ```json
   {
     "source": "clickstream_events",
     "version": 3,
     "previous_version": 2,
     "effective_date": "2024-07-01",
     "schema": {
       "type": "object",
       "required": ["event_id", "user_id", "event_type", "timestamp"],
       "properties": {
         "event_id": {"type": "string", "format": "uuid"},
         "user_id": {"type": "integer"},
         "event_type": {"type": "string", "enum": ["page_view", "click", "add_to_cart", "purchase"]},
         "timestamp": {"type": "string", "format": "date-time"},
         "page_url": {"type": "string"},
         "device_type": {"type": "string", "enum": ["desktop", "mobile", "tablet"]}
       }
     },
     "compatibility": "backward"
   }
   ```

2. Build the registry:
   ```python
   class SchemaRegistry:
       def register(self, source: str, schema: dict, version: int) -> None:
           """Store a new schema version"""

       def get_current(self, source: str) -> dict:
           """Get the latest schema for a source"""

       def get_version(self, source: str, version: int) -> dict:
           """Get a specific version"""

       def validate(self, source: str, record: dict) -> ValidationResult:
           """Validate a record against the current schema"""

       def check_compatibility(self, source: str, new_schema: dict) -> CompatibilityResult:
           """Check if new schema is compatible with the current one"""
   ```

3. Compatibility rules:
   - **Backward compatible**: new schema can read old data (adding optional fields is OK)
   - **Forward compatible**: old schema can read new data (removing optional fields is OK)
   - **Breaking**: renaming required fields, changing types, removing required fields
   ```python
   def check_backward_compatibility(self, old_schema: dict, new_schema: dict) -> List[str]:
       issues = []
       old_required = set(old_schema.get('required', []))
       new_required = set(new_schema.get('required', []))
       # New required fields that weren't in old schema = breaking
       new_requirements = new_required - old_required
       if new_requirements:
           issues.append(f"Breaking: new required fields: {new_requirements}")
       # Removed fields = might break consumers
       old_fields = set(old_schema['properties'].keys())
       new_fields = set(new_schema['properties'].keys())
       removed = old_fields - new_fields
       if removed:
           issues.append(f"Breaking: removed fields: {removed}")
       return issues
   ```

4. Schema drift detection — compare incoming data against registry at ingestion time:
   ```python
   def detect_drift(self, source: str, sample_records: List[dict]) -> DriftReport:
       schema = self.get_current(source)
       observed_fields = set()
       for record in sample_records:
           observed_fields.update(record.keys())
       expected_fields = set(schema['schema']['properties'].keys())
       new_fields = observed_fields - expected_fields
       missing_fields = expected_fields - observed_fields
       return DriftReport(new_fields=new_fields, missing_fields=missing_fields)
   ```

5. Store schemas in version-controlled files:
   ```
   schemas/
   ├── clickstream_events/
   │   ├── v1.json
   │   ├── v2.json
   │   └── v3.json  (current)
   ├── vendor_inventory/
   │   ├── v1.json
   │   └── v2.json  (current)
   ```

### Expected Output
- Registry: `tasks/84_schema_registry/registry.py`
- Schemas: `tasks/84_schema_registry/schemas/` (at least 3 sources)
- Compatibility checker: `tasks/84_schema_registry/compatibility.py`
- Demo: register schema v1, evolve to v2 (backward compatible), attempt v3 (breaking) → alert

### Common Mistakes
- Storing schemas only in code comments — no machine-readable validation possible
- Not versioning — you can't answer "what schema was active when this data was loaded?"
- Strict validation that rejects all unknown fields — makes adding fields a breaking change
- Not checking compatibility before deploying a new schema — the check is the whole point

### Stretch Goal
Integrate the schema registry with the dead letter queue (Task 82): when a record fails schema validation, route it to the DLQ with the schema violation details.

## Task 85: Implement a Data Warehouse Slowly Changing Dimension Type 3

### Business Context
The marketing team needs to know: "What was each user's *previous* segment, and what is their *current* segment?" SCD Type 2 tracks all historical versions but requires complex queries to get "previous." SCD Type 3 stores the current and previous values directly in the dimension row — simpler queries for a common use case.

### Objective
Implement SCD Type 3 for the `dim_users` table, storing both current and previous values for key attributes (segment, tier, email).

### Technical Requirements
- **Tools**: SQL, dbt
- **Skills**: SCD Type 3, dimension design, update patterns, trade-offs vs SCD2

### Dataset
NovaMart `dim_users` and historical user changes.

### Step-by-Step Guidance
1. Design the SCD3 table:
   ```sql
   CREATE TABLE dim_users_scd3 (
       user_key SERIAL PRIMARY KEY,
       user_id INTEGER UNIQUE,
       email VARCHAR(255),
       name VARCHAR(255),

       -- Current values
       current_segment VARCHAR(50),
       current_loyalty_tier VARCHAR(20),
       current_city VARCHAR(100),

       -- Previous values
       previous_segment VARCHAR(50),
       previous_loyalty_tier VARCHAR(20),
       previous_city VARCHAR(100),

       -- Change tracking
       segment_changed_at DATE,
       loyalty_tier_changed_at DATE,
       city_changed_at DATE,

       created_at TIMESTAMP,
       updated_at TIMESTAMP
   );
   ```

2. Build the update logic:
   ```sql
   -- When a user's segment changes:
   UPDATE dim_users_scd3
   SET
       previous_segment = current_segment,          -- move current to previous
       current_segment = 'VIP',                      -- set new current
       segment_changed_at = CURRENT_DATE,
       updated_at = NOW()
   WHERE user_id = 42 AND current_segment != 'VIP';  -- only if actually changed
   ```

3. Implement as a dbt model:
   ```sql
   -- models/marts/dim_users_scd3.sql
   {{
     config(materialized='table')
   }}

   WITH current_users AS (
       SELECT * FROM {{ ref('stg_users') }}
   ),
   existing AS (
       SELECT * FROM {{ this }}
   ),
   changes AS (
       SELECT
           c.user_id,
           c.email,
           c.name,
           c.segment AS new_segment,
           c.loyalty_tier AS new_tier,
           c.city AS new_city,
           e.current_segment AS old_segment,
           e.current_loyalty_tier AS old_tier,
           e.current_city AS old_city
       FROM current_users c
       LEFT JOIN existing e ON c.user_id = e.user_id
       WHERE e.user_id IS NULL  -- new users
          OR c.segment != e.current_segment
          OR c.loyalty_tier != e.current_loyalty_tier
          OR c.city != e.current_city
   )
   -- ... merge logic
   ```

4. Compare SCD2 vs SCD3:
   | Aspect | SCD2 | SCD3 |
   |--------|------|------|
   | Full history | Yes | Only current + previous |
   | Query complexity | JOIN on date range | Simple column access |
   | Storage | Grows with changes | Fixed per user |
   | Use case | Audit, any point-in-time | "What changed from?" |

5. Query examples:
   ```sql

   SELECT * FROM dim_users_scd3
   WHERE current_segment = 'VIP' AND previous_segment = 'Regular';

   -- Users who changed segments in the last 30 days
   SELECT * FROM dim_users_scd3
   WHERE segment_changed_at >= CURRENT_DATE - 30;
   ```

### Expected Output
- SCD3 model: `tasks/85_scd3/dim_users_scd3.sql`
- Update logic: `tasks/85_scd3/scd3_update.sql`
- Comparison doc: `tasks/85_scd3/scd2_vs_scd3.md`
- Query examples demonstrating the "previous vs current" use case

### Common Mistakes
- Moving current → previous when nothing actually changed (no change check)
- Losing the previous value on the second change (third version is lost forever — that's the trade-off, document it)
- Using SCD3 when full history is needed — it only tracks one level of history
- Not handling new users (previous values should be NULL, not empty strings)

Implement a hybrid SCD2+SCD3: full history in SCD2 for audit, with the SCD3 convenience columns materialized as a view on top.

## Task 86: Build a Data Pipeline Versioning System

### Business Context
You deployed a pipeline change that broke the marketing mart. You need to roll back to yesterday's version. But the pipeline code, the dbt models, the Airflow DAGs, and the config files are all in different places. Priya says: "We need pipeline versioning — the ability to deploy, roll back, and compare pipeline versions as atomic units."

### Objective
Build a pipeline versioning system that bundles pipeline code, models, configs, and dependencies into versioned releases that can be deployed and rolled back atomically.

### Technical Requirements
- **Tools**: Python, Git, YAML, shell scripting
- **Skills**: Release management, version control, atomic deployments, rollback procedures

### Dataset
NovaMart's pipeline codebase (all previous tasks).

### Step-by-Step Guidance
1. Define a pipeline release manifest:
   ```yaml
   # releases/v2.5.0/manifest.yaml
   version: "2.5.0"
   release_date: "2024-07-20"
   previous_version: "2.4.1"
   author: "your_name"
   description: "Add loyalty points pipeline, fix vendor inventory SLA"
   components:
     - name: vendor_inventory_etl
       type: airflow_dag
       path: dags/vendor_inventory_etl.py
       checksum: sha256:abc123...
     - name: dbt_models
       type: dbt_project
       path: dbt/
       checksum: sha256:def456...
     - name: pipeline_config
       type: config
       path: config/pipeline_config.yaml
       checksum: sha256:ghi789...
   migrations:
     - 003_add_loyalty_points.up.sql
   rollback_migrations:
     - 003_add_loyalty_points.down.sql
   ```

2. Build a release tool:
   ```python
   class ReleaseManager:
       def create_release(self, version: str, description: str) -> Release:
           """Snapshot current code, compute checksums, create manifest"""
           manifest = {
               'version': version,
               'components': self._snapshot_components(),
               'migrations': self._pending_migrations(),
           }
           self._write_manifest(version, manifest)
           self._tag_git(version)
           return Release(version, manifest)

       def deploy(self, version: str, env: str) -> DeployResult:
           """Deploy a specific version to an environment"""
           release = self.load_release(version)
           self._run_migrations(release.migrations)
           self._deploy_dags(release.components)
           self._deploy_config(release.components)
           self._record_deployment(version, env)

       def rollback(self, env: str) -> DeployResult:
           """Roll back to the previous version"""
           current = self.get_current_version(env)
           previous = current.previous_version
           self._run_migrations(current.rollback_migrations)
           return self.deploy(previous, env)

       def diff(self, version_a: str, version_b: str) -> List[str]:
           """Show what changed between two versions"""
   ```

3. Track deployment history:
   ```sql
   CREATE TABLE deployment_history (
       deploy_id SERIAL PRIMARY KEY,
       version VARCHAR(20),
       environment VARCHAR(20),
       deployed_at TIMESTAMP DEFAULT NOW(),
       deployed_by VARCHAR(100),
       status VARCHAR(20),  -- 'success', 'failed', 'rolled_back'
       rollback_of INTEGER REFERENCES deployment_history(deploy_id)
   );
   ```

4. Pre-deployment checks:
   - All tests pass
   - No breaking schema changes without migration
   - Config diff reviewed
5. Post-deployment verification:
   - Pipeline runs successfully
   - Row counts within expected range
   - SLA checks pass

### Expected Output
- Release manager: `tasks/86_pipeline_versioning/release_manager.py`
- Sample release manifest: `tasks/86_pipeline_versioning/releases/v1.0.0/manifest.yaml`
- Deployment history table
- Demo: create release, deploy, introduce bug, rollback

### Common Mistakes
- Deploying code without the matching migration — schema and code are out of sync
- No checksums — you can't verify that what's deployed matches what was released
- Rolling back code but not the database — migrations and code must move together
- No deployment record — "What version is running in prod?" should be one query away

### Stretch Goal
Add a canary deployment: deploy the new version to process 10% of data, compare results with the old version, automatically promote or rollback.

### Dataset

### Step-by-Step Guidance
1. Define circuit breaker checks:
   ```python
   class CircuitBreaker:
       def __init__(self, pipeline_name: str, checks: list):
           self.pipeline_name = pipeline_name
           self.checks = checks
           self.state = 'CLOSED'  # CLOSED=normal, OPEN=blocked, HALF_OPEN=testing

       def evaluate(self, data: pd.DataFrame) -> CircuitBreakerResult:
           failures = []
           for check in self.checks:
               result = check.evaluate(data)
               if not result.passed:
                   failures.append(result)

           if any(f.severity == 'critical' for f in failures):
               self.state = 'OPEN'
               return CircuitBreakerResult(action='HALT', failures=failures)
           elif failures:
               return CircuitBreakerResult(action='WARN', failures=failures)
           else:
               return CircuitBreakerResult(action='PROCEED')
   ```

2. Implement common checks:
   ```python
   class RowCountCheck:
       """Halt if row count drops >50% from previous load"""
       def evaluate(self, data):
           previous = get_previous_row_count(self.pipeline_name)
           current = len(data)
           if previous > 0 and current < previous * 0.5:
               return CheckResult(passed=False, severity='critical',
                   message=f'Row count dropped {current} vs previous {previous}')

   class NullRateCheck:
       """Halt if null rate exceeds threshold"""
       def evaluate(self, data):
           for col in self.critical_columns:
               null_rate = data[col].isna().mean()
               if null_rate > self.threshold:
                   return CheckResult(passed=False, severity='critical',
                       message=f'{col}: null rate {null_rate:.1%} > {self.threshold:.1%}')

   class ValueRangeCheck:
       """Halt if values are outside expected range"""

   class SchemaCheck:
       """Halt if schema has changed unexpectedly"""

   class EmptyDataCheck:
       """Halt if dataset is completely empty"""
   ```

3. Integrate into the pipeline, between transform and load:

           ValueRangeCheck(column='price', min=0, max=100000),

       is_current = FALSE

2. Build a micro-batch processor:
   ```python
   class MicroBatchProcessor:
       def __init__(self, batch_interval_seconds: int = 10):
           self.batch_interval = batch_interval_seconds
           self.buffer = []

       def add_event(self, event: dict):
           self.buffer.append(event)

       def should_flush(self) -> bool:
           if not self.buffer:
               return False
           oldest = min(e['processing_time'] for e in self.buffer)
           return (datetime.utcnow() - oldest).seconds >= self.batch_interval

       def flush(self) -> list:
           batch = self.buffer
           self.buffer = []
           return batch
   ```

3. Implement tumbling windows:
   ```python
   class TumblingWindow:
       def __init__(self, window_size_seconds: int = 60):
           self.window_size = window_size_seconds
           self.windows = defaultdict(list)

       def assign(self, event: dict) -> str:
           """Assign event to its window based on event_time (not processing_time)"""
           event_time = event['event_time']
           window_start = event_time.replace(
               second=0, microsecond=0
           ) - timedelta(seconds=event_time.second % self.window_size)
           window_key = window_start.isoformat()
           self.windows[window_key].append(event)
           return window_key

       def get_completed_windows(self, watermark: datetime) -> list:
           """Return windows whose end time is before the watermark"""
           completed = []
           for window_key, events in list(self.windows.items()):
               window_start = datetime.fromisoformat(window_key)
               window_end = window_start + timedelta(seconds=self.window_size)
               if window_end <= watermark:
                   completed.append((window_key, events))
                   del self.windows[window_key]
           return completed
   ```

4. Implement watermarks:
   ```python
   class WatermarkTracker:
       def __init__(self, max_lateness_seconds: int = 30):
           self.max_lateness = max_lateness_seconds
           self.max_event_time = None

       def update(self, event_time: datetime):
           if self.max_event_time is None or event_time > self.max_event_time:
               self.max_event_time = event_time

       @property
       def watermark(self) -> datetime:
           """Watermark = max event time - max allowed lateness"""
           if self.max_event_time is None:
               return datetime.min
           return self.max_event_time - timedelta(seconds=self.max_lateness)
   ```

5. Handle late data:
   ```python
   def process_event(event):
       if event['event_time'] < watermark_tracker.watermark:
           # Late event — window already closed
           route_to_late_data_handler(event)
       else:
           window.assign(event)
   ```

6. Aggregate each window:
   ```python

       return {
           'window': window_key,
           'event_count': len(events),
           'unique_users': len(set(e['user_id'] for e in events)),
           'purchases': sum(1 for e in events if e['event_type'] == 'purchase'),
           'page_views': sum(1 for e in events if e['event_type'] == 'page_view'),
       }
   ```

7. Run for 5 minutes, output window aggregates as they complete

- Windowing: `tasks/96_streaming_sim/windowing.py`

### Common Mistakes
- Using processing time for windows — events can arrive out of order; use event time
- No watermark — windows never close because you're always waiting for "one more late event"
- Dropping all late events — some systems update the window retroactively; document the trade-off

#### Phase 2: Build the Data Pipeline (4-6 hours)
3. Build micro-batch ingestion (runs every 1 minute):
   ```python
   class MicroBatchIngester:
       def __init__(self):
           self.watermark = get_last_watermark()

       def ingest_new_orders(self) -> int:
           new_orders = query("""
               SELECT * FROM orders
               WHERE updated_at > :watermark
               ORDER BY updated_at
           """, watermark=self.watermark)

           if new_orders:
               load_to_staging(new_orders)
               self.watermark = max(o['updated_at'] for o in new_orders)
               save_watermark(self.watermark)
           return len(new_orders)
   ```

4. Build the aggregation layer:
   ```sql
   -- real_time_order_summary (materialized view or table)
   CREATE TABLE real_time_order_summary AS
   SELECT
       DATE_TRUNC('minute', order_date) AS minute_bucket,
       COUNT(*) AS order_count,
       SUM(total_amount) AS revenue,
       COUNT(DISTINCT user_id) AS unique_customers,
       AVG(total_amount) AS avg_order_value
   FROM orders
   WHERE order_date >= CURRENT_DATE
   GROUP BY 1;
   ```

5. Build comparison metrics:
   ```sql
   -- Today vs yesterday at the same time
   SELECT
       today.minute_bucket,
       today.revenue AS revenue_today,
       yesterday.revenue AS revenue_yesterday,
       ROUND((today.revenue - yesterday.revenue) / NULLIF(yesterday.revenue, 0) * 100, 1) AS pct_change
   FROM real_time_order_summary today
   LEFT JOIN real_time_order_summary_yesterday yesterday
       ON today.minute_bucket::time = yesterday.minute_bucket::time;
   ```

6. Product ranking:
   ```sql
   SELECT
       p.product_name,
       p.category,
       COUNT(*) AS units_sold,
       SUM(oi.unit_price * oi.quantity) AS revenue
   FROM order_items oi
   JOIN products p ON oi.product_id = p.product_id
   JOIN orders o ON oi.order_id = o.order_id
   WHERE o.order_date >= NOW() - INTERVAL '1 hour'
   GROUP BY p.product_name, p.category
   ORDER BY revenue DESC
   LIMIT 10;
   ```

### Common Mistakes
- Trying to build actual real-time (Kafka/Flink) — micro-batch is perfect for a Junior capstone
- Dashboard with no auto-refresh — "real-time" means auto-updating
- No comparison metrics — absolute numbers are useless without context (vs yesterday, vs average)
- Over-scoping — MVP first, polish later. A working dashboard with 4 KPIs beats a broken one with 20
