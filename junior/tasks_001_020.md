## Task 2: Fix the Broken Revenue Report

### Business Context
The finance team's daily revenue dashboard shows $0 for several days last month. The CEO noticed. Priya assigns you the ticket: "Investigate and fix the revenue report query."

### Objective
Debug and rewrite the revenue report query to correctly calculate daily revenue, handling cancelled orders, partial refunds, and NULL payment amounts.

### Technical Requirements
- **Tools**: SQL (PostgreSQL)
- **Skills**: JOINs, WHERE vs HAVING, CASE statements, date functions, NULL handling (COALESCE)

### Dataset
Same NovaMart database from Task 1. Ensure `orders`, `order_items`, and `payments` tables are populated.

### Step-by-Step Guidance
1. The "broken" query is:
   ```sql
   SELECT order_date::DATE, SUM(total_amount) as revenue
   FROM orders
   GROUP BY order_date::DATE
   ORDER BY order_date::DATE;
   ```
2. Identify why this is wrong (hint: it doesn't exclude cancelled/returned orders, doesn't account for failed payments)
3. Write a corrected query that:
   - Only counts orders with status `confirmed`, `shipped`, or `delivered`
   - Joins to `payments` and only counts `success` payments
   - Handles NULLs in `total_amount` with COALESCE
   - Fills in zero-revenue days using a date series
4. Compare old vs new results

### Expected Output
- SQL files: `tasks/02_revenue_fix/broken_query.sql` and `tasks/02_revenue_fix/fixed_query.sql`
- A short write-up explaining what was wrong and how you fixed it

### Common Mistakes
- Forgetting to handle days with zero orders (they disappear from GROUP BY)
- Double-counting revenue when an order has multiple payments (retries)
- Not filtering on payment status

### Stretch Goal
Generate a date-spine table and LEFT JOIN to ensure every day appears in the report, even with $0 revenue.

## Task 12: Slowly Changing Data — Handling Updates

### Business Context
Users update their profiles — changing names, email addresses, and shipping addresses. The current system just overwrites the old data. But the analytics team says: "We need to know what a user's address was *at the time of their order*, not what it is today." You need to handle slowly changing dimensions.

### Objective
Implement a Type 2 Slowly Changing Dimension (SCD2) for the `users` table — track historical changes with effective dates.

### Technical Requirements
- **Tools**: SQL (PostgreSQL), Python
- **Skills**: SCD concepts, MERGE/UPSERT logic, date range tracking, surrogate keys

### Dataset
Create a simulated user change feed:
```csv
user_id,full_name,email,country,updated_at
1,Alice Smith,alice@example.com,US,2024-01-01 00:00:00
2,Bob Jones,bob@example.com,US,2024-01-01 00:00:00
1,Alice Johnson,alice.johnson@example.com,US,2024-03-15 10:00:00
2,Bob Jones,bob@example.com,CA,2024-06-01 08:00:00
3,Charlie Brown,charlie@example.com,UK,2024-06-15 12:00:00
1,Alice Johnson,alice.johnson@example.com,CA,2024-07-01 09:00:00
```

### Step-by-Step Guidance
1. Create an SCD2 dimension table:
   ```sql
   CREATE TABLE dim_users (
       user_key SERIAL PRIMARY KEY,  -- surrogate key
       user_id INTEGER,              -- natural key
       full_name VARCHAR(255),
       email VARCHAR(255),
       country VARCHAR(50),
       effective_from TIMESTAMP,
       effective_to TIMESTAMP,       -- NULL = current record
       is_current BOOLEAN DEFAULT TRUE
   );
   ```
2. Write a Python script that processes the change feed:
   - For new users: INSERT a new row with `effective_from = updated_at`, `effective_to = NULL`, `is_current = TRUE`
   - For changed users: UPDATE the current row's `effective_to` and `is_current = FALSE`, then INSERT a new current row
   - For unchanged users: skip
3. Verify: user_id=1 should have 3 rows (3 versions), user_id=2 should have 2 rows

### Expected Output
- SQL DDL: `tasks/12_scd2/create_dim_users.sql`
- Python script: `tasks/12_scd2/process_scd2.py`
- Query showing all historical versions for user_id=1

### Common Mistakes
- Comparing all columns to detect changes — must exclude `updated_at` from comparison
- Not handling the case where a "new" record is identical to the current one (no actual change)
- Using `effective_to = '9999-12-31'` instead of NULL — NULL is cleaner and avoids Y10K bugs

### Stretch Goal
Implement this in dbt using the `dbt-utils` or `dbt_slowly_changing_dimensions` package (snapshot).

## Task 20: End-of-Month Reconciliation Pipeline

### Business Context
End of month. Finance needs to reconcile: total orders in our system vs total charges in Stripe vs total shipments logged. If the numbers don't match, something is wrong — potentially lost revenue or phantom orders. This is a critical monthly process.

### Objective
Build a reconciliation pipeline that cross-references orders, payments, and shipping data, identifies mismatches, and produces an exception report.

### Technical Requirements
- **Tools**: SQL, Python, pandas
- **Skills**: FULL OUTER JOINs, set difference operations, reconciliation patterns, exception reporting

### Dataset
NovaMart `orders`, `payments`, `shipping` tables. Introduce intentional mismatches:
```sql
-- Order with no payment record
INSERT INTO orders (order_id, user_id, order_date, status, total_amount)
VALUES (99901, 1, '2024-07-15', 'confirmed', 150.00);

-- Payment with no matching order (orphan)
INSERT INTO payments (payment_id, order_id, payment_date, amount, method, status)
VALUES (99801, 99999, '2024-07-20', 75.00, 'credit_card', 'success');

-- Order amount != payment amount (mismatch)
INSERT INTO orders (order_id, user_id, order_date, status, total_amount)
VALUES (99902, 2, '2024-07-22', 'delivered', 200.00);
INSERT INTO payments (payment_id, order_id, payment_date, amount, method, status)
VALUES (99802, 99902, '2024-07-22', 180.00, 'credit_card', 'success');

-- Shipped order with no shipping record
INSERT INTO orders (order_id, user_id, order_date, status, total_amount)
VALUES (99903, 3, '2024-07-25', 'shipped', 99.00);
```

### Step-by-Step Guidance
1. **Orders vs Payments**:
   - FULL OUTER JOIN orders and payments on `order_id`
   - Flag: orders without payments, payments without orders, amount mismatches
2. **Orders vs Shipping**:
   - For orders with status `shipped` or `delivered`, verify a shipping record exists
   - Flag: shipped orders without tracking, shipping records for cancelled orders
3. **Summary Stats**:
   - Total orders: X, Total payments: Y, Match rate: Z%
   - Total mismatches by type
4. **Exception Report**:
   - Generate a CSV/Markdown with all exceptions: `order_id`, `mismatch_type`, `order_amount`, `payment_amount`, `details`
5. Package this as a Python script that could run monthly via Airflow

### Expected Output
- Reconciliation queries: `tasks/20_reconciliation/reconciliation.sql`
- Pipeline script: `tasks/20_reconciliation/reconcile.py`
- Exception report: `tasks/20_reconciliation/exceptions_2024_07.csv`
- Summary: `tasks/20_reconciliation/summary.md`

### Common Mistakes
- Using INNER JOIN (misses the mismatches you're looking for!)
- Not handling 1:many relationships — one order can have multiple payments (split payment, retry)
- Treating all mismatches as equal — an orphan payment is more urgent than a $0.01 rounding difference
- Not categorizing exceptions — the finance team needs to know *what type* of problem each row is

### Stretch Goal
Add threshold-based alerting: if match rate drops below 99%, automatically flag as critical. Add a tolerance for rounding differences (< $0.05 = OK).
