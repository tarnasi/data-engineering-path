## Task 41: dbt Incremental Models

### Business Context
The `daily_business_summary` dbt model (Task 11) does a full rebuild every run — it scans the entire `orders` table going back 2 years. As data grows, this takes longer every day. Priya says: "Convert it to an incremental model. Only process new data on each run."

### Objective
Refactor the `daily_business_summary` dbt model to use dbt's incremental materialization, processing only new or changed records since the last run.

### Technical Requirements
- **Tools**: dbt-core (incremental materialization)
- **Skills**: dbt incremental models, `is_incremental()` macro, unique_key, merge strategies

### Dataset
NovaMart dbt project from Tasks 11/16.

### Step-by-Step Guidance
1. Change the model's materialization:
   ```sql
   {{
     config(
       materialized='incremental',
       unique_key='report_date',
       incremental_strategy='delete+insert'
     )
   }}

   SELECT
       o.order_date::date AS report_date,
       COUNT(DISTINCT o.order_id) AS order_count,
       SUM(p.amount) AS total_revenue,
       ...
   FROM {{ ref('stg_orders') }} o
   JOIN {{ ref('stg_payments') }} p ON o.order_id = p.order_id

   {% if is_incremental() %}
   WHERE o.order_date >= (SELECT MAX(report_date) - INTERVAL '3 days' FROM {{ this }})
   {% endif %}

   GROUP BY 1
   ```
2. The `- INTERVAL '3 days'` lookback handles late-arriving data (from Task 32)
3. Run: `dbt run --select daily_business_summary` (full build first time)
4. Add new orders, run again — verify only recent days reprocessed
5. Test: `dbt run --full-refresh` for forced rebuilds
6. Add a test: `dbt test` to verify no duplicate `report_date` values

### Expected Output
- Updated model: `models/marts/daily_business_summary.sql` (incremental)
- Updated schema YAML with tests
- Log output showing incremental vs full-refresh row counts

### Common Mistakes
- Forgetting the lookback window — only processing `> MAX(report_date)` misses late-arriving data
- Using `merge` strategy without a proper `unique_key` — creates duplicates
- Not testing with `--full-refresh` — incremental models must work in both modes
- Setting the filter on `loaded_at` when it should be on `order_date` (business date vs processing date)

### Stretch Goal
Add a `merge` strategy incremental model for `customer_segments` where the unique key is `user_id` and changed fields get updated in place.

## Task 42: dbt Macros and DRY Transformations

### Business Context
You notice the same SQL pattern repeated across 8 dbt models: converting cents to dollars, masking PII, calculating date differences. Priya reviews your PR and says: "Don't repeat yourself. Write macros."

### Objective
Write reusable dbt macros to eliminate duplicated SQL logic across your models.

### Technical Requirements
- **Tools**: dbt-core (Jinja macros)
- **Skills**: Jinja templating, dbt macros, macro arguments, macro libraries

### Dataset
NovaMart dbt project — identify 3+ patterns that repeat across models.

### Step-by-Step Guidance
1. Create `macros/cents_to_dollars.sql`:
   ```sql
   {% macro cents_to_dollars(column_name) %}
       ({{ column_name }} / 100.0)::numeric(10,2)
   {% endmacro %}
   ```
2. Create `macros/mask_email.sql`:
   ```sql
   {% macro mask_email(column_name) %}
       CASE
           WHEN {{ column_name }} IS NOT NULL
           THEN '***' || SUBSTRING({{ column_name }} FROM POSITION('@' IN {{ column_name }}))
           ELSE NULL
       END
   {% endmacro %}
   ```
3. Create `macros/date_diff_days.sql`:
   ```sql
   {% macro date_diff_days(start_date, end_date) %}
       EXTRACT(DAY FROM ({{ end_date }}::timestamp - {{ start_date }}::timestamp))
   {% endmacro %}
   ```
4. Create a conditional macro:
   ```sql
   {% macro safe_divide(numerator, denominator, default=0) %}
       CASE
           WHEN {{ denominator }} = 0 OR {{ denominator }} IS NULL THEN {{ default }}
           ELSE ({{ numerator }}::numeric / {{ denominator }}::numeric)
       END
   {% endmacro %}
   ```
5. Refactor existing models to use these macros:
   ```sql
   -- Before
   SELECT (amount / 100.0)::numeric(10,2) AS amount_dollars
   -- After
   SELECT {{ cents_to_dollars('amount') }} AS amount_dollars
   ```
6. Run `dbt run` to verify nothing breaks

### Expected Output
- Macros: `macros/cents_to_dollars.sql`, `mask_email.sql`, `date_diff_days.sql`, `safe_divide.sql`
- At least 3 models refactored to use the macros
- `dbt run` passing with no changes to output data

### Common Mistakes
- Creating macros for one-off logic (over-engineering) — only macro repeated patterns
- Not quoting column names properly in macros — breaks when column names have spaces or reserved words
- Macros with side effects (modifying state) — macros should be pure SQL generators
- Forgetting to document macros — teammates won't know they exist

### Stretch Goal
Write a `generate_schema_name` macro that auto-routes models to different schemas based on their folder (staging → `stg`, marts → `analytics`).

## Task 44: Spark Transformations and Aggregations

### Business Context
The data science team wants a daily feature table for their recommendation model: per-user aggregated metrics from clickstream data. This needs to run on the full 500M-event dataset, so Spark is mandatory.

### Objective
Write a PySpark pipeline that computes per-user daily aggregated features from clickstream data — event counts, session metrics, and behavioral signals.

### Technical Requirements
- **Tools**: PySpark
- **Skills**: groupBy, agg, window functions in Spark, pivot, multiple aggregations

### Dataset
Clickstream Parquet files from Task 43 (or generate larger dataset).

### Step-by-Step Guidance
1. Read from partitioned Parquet:
   ```python
   df = spark.read.parquet("output/clickstream_parquet/")
   ```
2. Compute per-user daily features:
   ```python
   from pyspark.sql import functions as F

   user_daily = df.groupBy("user_id", "event_date").agg(
       F.count("event_id").alias("total_events"),
       F.countDistinct("session_id").alias("unique_sessions"),
       F.sum(F.when(F.col("event_type") == "page_view", 1).otherwise(0)).alias("page_views"),
       F.sum(F.when(F.col("event_type") == "add_to_cart", 1).otherwise(0)).alias("cart_adds"),
       F.sum(F.when(F.col("event_type") == "purchase", 1).otherwise(0)).alias("purchases"),
       F.min("timestamp").alias("first_event"),
       F.max("timestamp").alias("last_event"),
   )
   ```
3. Add session-level features:
   - Average events per session
   - Session duration (last event - first event per session)
4. Add conversion flag: `has_purchase = purchases > 0`
5. Compute user-level rolling features (Spark window functions):
   ```python
   from pyspark.sql.window import Window

   user_window = Window.partitionBy("user_id").orderBy("event_date").rowsBetween(-6, 0)
   user_daily = user_daily.withColumn(
       "events_7d_avg", F.avg("total_events").over(user_window)
   )
   ```
6. Write output to Parquet: `output/user_daily_features/`

### Expected Output
- Feature pipeline: `tasks/44_spark_aggregations/user_features.py`
- Feature table with columns: user_id, event_date, total_events, unique_sessions, page_views, cart_adds, purchases, avg_session_duration, events_7d_avg, has_purchase
- Row count and sample output

### Common Mistakes
- Doing groupBy + join + groupBy + join chains instead of computing multiple aggregations at once
- Not caching intermediate DataFrames that are reused — Spark recomputes them each time
- Using `.toPandas()` for "quick" operations on a 500M-row DataFrame — OOM crash
- Ignoring null user_ids — they create a massive "anonymous" user group that distorts metrics

### Stretch Goal
Add a "user lifetime" feature table: total orders, total spend, days since first/last order, favorite category. Join Spark clickstream features with PostgreSQL order data.

No one told you about the change. Priya says: "We need data contracts. Upstream teams must declare their schema, and changes must be coordinated."

### Objective
Design and implement a data contract system — formal schema definitions that producers and consumers agree on, with validation that catches breaking changes before they hit production.

### Technical Requirements
- **Tools**: Python, JSON Schema or Pydantic, YAML
- **Skills**: Schema definition, schema evolution rules, backwards compatibility, contract testing

### Dataset
Clickstream event schema — current version and the new version from the mobile team.

### Step-by-Step Guidance
1. Define the data contract as a YAML/JSON schema:
   ```yaml
   # contracts/clickstream_events_v1.yaml
   contract:
     name: clickstream_events
     version: 1
     owner: mobile-team
     consumers: [data-engineering, data-science]
     sla:
       freshness: 1 hour
       completeness: 99.5%
     schema:
       fields:
         - name: event_id
           type: string
           required: true
         - name: user_id
           type: integer
           required: false
         - name: event_type
           type: string
           required: true
           allowed_values: [page_view, search, add_to_cart, checkout_start, purchase]
         - name: page_url
           type: string
           required: true
         - name: timestamp
           type: timestamp
           required: true
           format: "ISO8601"
         - name: context
           type: object
           fields:
             - name: device
               type: string
               required: true
             - name: browser
               type: string
               required: true
   ```
2. Build a contract validator:
   ```python
   class ContractValidator:
       def validate_record(self, record: dict, contract: dict) -> List[str]:
           """Return list of violations"""

       def validate_schema_evolution(self, old_contract: dict, new_contract: dict) -> dict:
           """Check if schema change is backwards compatible"""
           # Breaking: removing a required field, renaming a field, changing a type
           # Non-breaking: adding optional field, adding allowed_values
   ```
3. Validate the mobile team's proposed change:
   - Renamed `page_url` → `screen_name`: **BREAKING** (consumers rely on `page_url`)
   - Added `screen_resolution`: **NON-BREAKING** (new optional field)
4. Integrate validation into the ingestion pipeline: reject records that violate the contract
5. Write documentation: "How to request a schema change" (must update contract, get consumer sign-off)

- Validator: `tasks/45_data_contracts/contract_validator.py`
- Schema evolution report showing which changes are breaking vs non-breaking

### Common Mistakes
- Only validating field names, not types — a `user_id` changing from `int` to `string` is breaking
- No versioning — contracts must have versions so consumers can migrate at their own pace
- Making all fields required — overly strict contracts break constantly
- Not involving consumers in contract changes — data engineering finds out after the deploy

## Task 46: Build a Slowly Changing Dimension in dbt

### Business Context
Remember the SCD2 implementation from Task 12? Priya says: "Good work, but that was raw Python. Let's do it properly in dbt using snapshots. This way it's version-controlled, tested, and runs with one command."

### Objective
Implement dbt snapshots to automatically track changes to the `users` and `products` dimension tables using dbt's built-in SCD2 functionality.

### Technical Requirements
- **Tools**: dbt-core (snapshots)
- **Skills**: dbt snapshots, snapshot strategies (timestamp vs check), snapshot testing

### Dataset
NovaMart `users` and `products` tables with simulated changes over time.

### Step-by-Step Guidance
1. Create a snapshot for users: `snapshots/snap_users.sql`
   ```sql
   {% snapshot snap_users %}

   {{
     config(
       target_schema='snapshots',
       unique_key='user_id',
       strategy='timestamp',
       updated_at='updated_at',
     )
   }}

   SELECT * FROM {{ source('novamart', 'users') }}

   {% endsnapshot %}
   ```
2. Create a snapshot for products: `snapshots/snap_products.sql`
   ```sql
   {% snapshot snap_products %}

   {{
     config(
       target_schema='snapshots',
       unique_key='product_id',
       strategy='check',
       check_cols=['name', 'price', 'category', 'is_active'],
     )
   }}

   SELECT * FROM {{ source('novamart', 'products') }}

   {% endsnapshot %}
   ```
3. Run `dbt snapshot` — this creates the snapshot tables with SCD2 columns:
   - `dbt_scd_id`, `dbt_updated_at`, `dbt_valid_from`, `dbt_valid_to`
4. Simulate changes:
   ```sql
   UPDATE users SET country = 'CA', updated_at = NOW() WHERE user_id = 1;
   UPDATE products SET price = 39.99 WHERE product_id = 5;
   ```
5. Run `dbt snapshot` again — verify new rows appear with updated validity ranges
6. Use the snapshot in a downstream model:
   ```sql
   -- Get the user's country AT THE TIME of their order
   SELECT o.order_id, o.order_date, su.country
   FROM {{ ref('stg_orders') }} o
   JOIN {{ ref('snap_users') }} su
     ON o.user_id = su.user_id
     AND o.order_date >= su.dbt_valid_from
     AND (o.order_date < su.dbt_valid_to OR su.dbt_valid_to IS NULL)
   ```

### Expected Output
- Snapshots: `snapshots/snap_users.sql`, `snapshots/snap_products.sql`
- Downstream model using the snapshot: `models/marts/orders_with_user_history.sql`
- Output showing historical records for a changed user

### Common Mistakes
- Using `strategy='timestamp'` without an `updated_at` column on the source → snapshot never detects changes
- Using `strategy='check'` with `check_cols='all'` → irrelevant column changes trigger new snapshots
- Joining snapshots without the date-range filter → Cartesian product explosion
- Running snapshots too infrequently → large batches of changes get a single timestamp

### Stretch Goal
Add snapshot tests: verify that for each `unique_key`, exactly one record has `dbt_valid_to IS NULL` (the current record).

## Task 48: Handling Schema Drift in Pipelines

### Business Context
Vendor B suddenly starts sending a CSV with 2 extra columns (warehouse_location, expiry_date) and changes the column order. Your rigid pipeline crashes. Vendor C removes a column they said was "required." This happens constantly. Priya says: "Build pipelines that handle schema drift gracefully."

### Technical Requirements
- **Tools**: Python (pandas), PySpark, dbt
- **Skills**: Schema drift handling, defensive programming, schema inference, column mapping

### Dataset
Three versions of vendor B's file simulating schema drift:


### Step-by-Step Guidance

   ```python
   class SchemaReconciler:
       def __init__(self, expected_schema: dict):
           self.required_columns = expected_schema['required']
           self.optional_columns = expected_schema['optional']
           self.column_types = expected_schema['types']

       def reconcile(self, df: pd.DataFrame) -> tuple[pd.DataFrame, dict]:
           """
           Returns (reconciled_df, drift_report)
           - Maps columns regardless of order
           - Adds missing optional columns with defaults
           - Drops unexpected columns (log them)
   ```

   ```yaml
       quantity_available: integer
       unit_cost: float
   ```

- Drift report output

- Failing on ANY schema change — real-world data sources change constantly
- Silently accepting all changes — new columns might overwrite existing data
- Relying on column position instead of column name — position changes break everything
- Not alerting on drift — auto-handling is good, but the team should still know about changes

- **Tools**: PySpark, Spark UI
- **Skills**: Repartition vs coalesce, cache vs persist, broadcast joins, shuffle optimization, Spark UI

### Dataset

### Step-by-Step Guidance
1. **Understand current performance**: check the Spark UI at `localhost:4040`
   - How many stages? How many tasks?
   - Where are the shuffles? How much data is shuffled?
2. **Optimize partitioning**:
   ```python
   # Before: default partition count (200!) after a groupBy
   df.groupBy("user_id").agg(...)

   # After: set appropriate partition count
   spark.conf.set("spark.sql.shuffle.partitions", 50)  # based on data size
   ```
3. **Use broadcast joins** for small dimension tables:
   ```python
   from pyspark.sql.functions import broadcast

   # products table is small (~10K rows) — broadcast it
   df_enriched = df_events.join(broadcast(df_products), "product_id")
   ```
4. **Cache intermediate DataFrames** that are reused:
   ```python
   df_cleaned = df_raw.filter(...).withColumn(...)
   df_cleaned.cache()  # materializes in memory

   # Used in multiple downstream operations
   result1 = df_cleaned.groupBy("user_id").agg(...)
   result2 = df_cleaned.groupBy("event_date").agg(...)

   df_cleaned.unpersist()  # free memory when done
   ```
5. **Avoid unnecessary shuffles**:
   - Use `coalesce()` instead of `repartition()` when reducing partitions
   - Pre-partition data before multiple groupBys on the same key
6. **Measure improvement**: before vs after execution time and shuffle data volume

- Performance comparison: original vs optimized (time, shuffle volume, stages)
- Documentation of each optimization and why it helped

- Caching everything — memory is limited; only cache DataFrames used multiple times
- Setting `shuffle.partitions` too low — tasks get too large, OOM; too high — overhead from scheduling
- Using `repartition()` before `write` when `coalesce()` is sufficient (repartition = full shuffle)

## Task 60: Junior Final Capstone — NovaMart Q3 Analytics Pipeline

### Business Context
It's the end of Q3. Leadership wants a comprehensive quarterly analytics pipeline that produces all key reports for the board meeting: revenue summary, customer growth, product performance, operational metrics, and data quality scores. This is your final Junior capstone — demonstrate everything you've learned.

### Objective
Build a complete, production-grade quarterly analytics pipeline that ingests data, transforms it through your dbt models, runs quality checks, produces executive-ready output tables, and monitors itself.

### Technical Requirements
- **Tools**: Everything you've learned — Python, SQL, dbt, Airflow, PySpark (optional), Great Expectations
- **Skills**: End-to-end pipeline design, orchestration, quality, monitoring, documentation

### Dataset
All NovaMart data — this pipeline should work on 3 months of data.

### Step-by-Step Guidance
1. **Define the quarterly report requirements**:
   - **Revenue**: monthly revenue, MoM growth, revenue by category, AOV trends
   - **Customers**: new vs returning, segment distribution, churn rate
   - **Products**: top sellers, underperformers, category mix
   - **Operations**: order fulfillment time, shipping success rate, return rate
   - **Data Quality**: table freshness, test pass rates, incident count
2. **Build the pipeline**:
   ```
   [INGEST] → [STAGE] → [TRANSFORM] → [QUALITY] → [REPORT TABLES] → [NOTIFY]
   ```
3. **Create dbt models** for each report section:
   - `models/reports/quarterly_revenue.sql`
   - `models/reports/quarterly_customers.sql`
   - `models/reports/quarterly_products.sql`
   - `models/reports/quarterly_operations.sql`
4. **Airflow DAG**: quarterly schedule (or manually triggered for Q3)
5. **Quality gates**: if data quality checks fail, don't produce the report (route to alert)
6. **Output**: final report tables readable by BI tools
7. **Documentation**: README explaining the pipeline, inputs, outputs, how to run, how to troubleshoot
8. **Monitoring**: freshness, run duration, row counts

### Expected Output
- dbt models: `models/reports/` (4+ models)
- Airflow DAG: `tasks/60_quarterly_pipeline/dags/quarterly_analytics_dag.py`
- Quality checks integrated
- Documentation: `tasks/60_quarterly_pipeline/README.md`
- Sample output for Q3 2024

### Common Mistakes
- Building the whole thing then testing — build and test each component incrementally
- Not defining "done" — agree on output columns before writing code
- Skipping data quality because "it's a one-time report" — quarterly reports to the board must be accurate
- Not documenting how to run it — the next person (or future you) won't know how
